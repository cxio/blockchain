對於一個開放的公共區塊鏈繫統來說，參與者是完全自由的，沒有激勵手段，自發的參與不可能形成。

在Bitcoin繫統中，激勵的手段有鑄幣奬勵和交易費獲利。鑄幣就是鑄造者可以按規定的數量憑空生成比特幣，這也是比特幣的來源。交易費是交易被納入區塊需要支付的費用，類似銀行轉賬的手續費。兩種收益構成了Bitcoin繫統運行的經濟支撐。

另外有一些區塊鏈繫統，當鑄幣到一定總量後，繫統以維持一個適度的通脹率緩慢發行新幣，這樣通脹鑄幣就和交易費一起成為繫統長期運行的驅動力。還有一些繫統僅以通脹鑄幣為驅動力，免除了用戶的交易費。但無論如何，繫統都需要經濟上的驅動因子，這就是公鏈上不可或缺的激勵機製。



## 原始鑄幣

類似於Bitcoin，本設計中的鑄幣是在漫長的時間延續中逐步發行。規則如下：

- 原始幣量：50幣/塊（致敬Bitcoin）。
- 出塊間隔：6分鈡。這是一個固定值。
- 遞減算法：逐年遞減10%。卽後一年是前一年的90%，精確到幣單位（而不是聰）。
- 發行時長：22年。精度單位提昇消除了後期微量鑄幣時期。
- 總發行量：34,889,078幣。

## 通脹鑄幣

通脹鑄幣是在原始鑄幣結束之後，繫統繼續以一個微小的通脹率長期發行新幣。通脹鑄幣有兩種方式：

1. 按鑄造者所屬交易的幣權依特定的利率生成新幣，如PoS中的利息。這裏可稱之為利率通脹。
2. 按固定的通脹率（通常可人為調整）生成新幣，與幣權多少無關（如EOS）。這裏可稱之為恆定通脹。

利率通脹的模式下，因為幣權不穩定會導致新幣的生產不穩定。恆定通脹模式下，通脹率基本維持穩定狀況。兩種方式有着自己的設計邏輯，各有優缺點，前者可以鼓勵古老存幣進入花費，而後者則沒有這樣的驅動力。

考慮簡單性和穩定性，本設計采用恆定通脹模式，通脹率可以由鑄造者在一個範圍內微調設定。


### 通脹繫數

鑄造者可以在Coinbase交易中設置一個通脹繫數來影響通脹率。通脹繫數是一個 `[0-100]` 的數值，表達年通脹率在 `0 ~ 4%` 範圍內的變化幅度，卽：`年通脹率 = x/100 * 4%`，x 卽為通脹繫數。繫統長期的年通脹率建議為 `1%`（x=25）。

在Coinbase交易的配置參數中，x值用1字節來表達，當設置 `x=127` 的時候表示無配置，這是該值的默認狀態。

繫統每30天自動統計一次，取之前 `7200個` 區塊裏有效設置的平均值計算本月通脹率。統計段中的區塊需要有 `2/3` 以上參與脩改才有效。


### 通縮機製

單純隻有通脹並不是一個邏輯完整的設計，因此**交易費可銷毀**實際上可以成為一個通縮設計。

交易費來源於已有的鑄幣，如果交易費被部分或全部地銷毀了，就會出現通縮的效果。類似於通脹繫數的設計，交易費的銷毀率也是用一個 `[0-100]` 的數值來表達，表示交易費銷毀的百分比（註：`迴收率 = 100 - 銷毀率`）。值 `127` 是一個特殊值，表示不設置。

同樣的，繫統每30天自動統計一次，取之前 `7200個` 區塊裏有效設置的平均值作為本月交易費的銷毀率，同樣參與脩改者需要超過 `2/3` 才有效。


### 小結：礦工的收入來源

1. 原始鑄幣：繫統基礎幣量，憑空生成。
2. 通脹鑄幣：依某種憑據計算而來，憑空生成。
3. 服務收費：卽交易費，由區塊鏈用戶支付。來源於已有的鑄幣，如果不是100%迴收就創建了通縮。



## 公共服務的激勵

公共服務分離出了區塊鏈繫統的基本需求，它們應當參與到區塊鏈的利益分配裏去，這是一個順理成章的邏輯。但這些公共服務是P2P的去中心化模式，一個邏輯嚴密的利益分配機製可能很難實現，因此這裏采用了一個鬆散的利益報償設計：**鑄造者收益的 `50%` 分配給公共服務**。


### 鑄幣交易的結構

區塊的鑄造收益在Coinbase交易裏體現，這是一個特殊的交易，包含了原始鑄幣、交易費提取以及後期的通脹鑄幣。與Bitcoin不同，本設計中的Coinbase交易還擔當了對外奬勵的規則實現者。交易的基本結構如下：

```go
Coinbase: {
    Version   int32         // 交易數據格式版本
    Timestamp uint64        // 交易時間戳。固定地與區塊創建時間相同。
    Minter    MintLocale    // 鑄造者引用。結構：4+32（區塊高度+交易ID）
    Config    ConfSet       // 鑄造配置集。鑄造者影響繫統的參數區（如激勵配比、通脹繫數等）
    // 統計點區塊包含
    ConfVal   ConfSet       // 鑄造配置結果區，階段性記錄
    CoinTotal int64         // 截止當前代幣存量（含通脹鑄幣和通縮銷毀）

    // 收益、激勵相關字段
    Coins     uint64        // 收益總額（鑄幣、交易費、權益金、兌奬截留）。
    CoinSelf  [2]*TxOut     // 鑄造收益（鑄造者，分成者）。約束：總額的50%。
    AwardTo   [3]*AwardOut  // 外部奬勵（Findings, Blockqs, Archives）。約束：總額的50%。
    DrawSlot  [90]byte      // 兌奬槽（3*240 = 720位）

    // 擇優池憑證
    // 頂端3名不參選候選者
    Meriter   [3]MintLocale // 3位候選者引用
    MeritSig  [4]SigMinter  // 3位候選者和鑄造者簽名數據
}
```

**說明：**
對公共服務的外部奬勵配比應該在繫統中硬編碼明確下來，這裏設計為按總額的50%平分。各個不同服務的內在配比由 `Config` 字段動態配置，如前所述，這些動態配置是一些受限的微調值，並且要求脩改配置的鑄造者達到一定數量才有效。


### 可能隨意的局部評估

對公共服務節點的奬勵需要一種抽奬機製，但公共服務是一個開放的P2P網絡。如果讓服務器提供某種證明來參與競爭，就需要創建一套防止作弊的可信機製，而這恰恰是區塊鏈本身要解決的問題。設計一套半可靠的方案旣復雜成本也高，因此，這裏的做法十分簡單：

**區塊的鑄造者可以隨意選擇獲奬者。沒有抽奬！**

也就是說，鑄造者可以選擇自己開設的服務，或者它們甚至都不是服務。乍一看貌似很荒唐，但其實並非如此。

奬勵的選擇是沒有限製的，但兌奬不是。獲奬者並不能簡單地把奬金拿走，奬金的提取需要得到後續鑄造者的認可。兌奬規則為：在後續240個區塊內，至少需要得到一位其他鑄造者的確認，確認數越多，提取的比例就越大。具體如下：

```go
確認數  兌現比例
------------------
1       20%
2       40%
3       60%
4       80%
5       100%
```

後續鑄造者對前期奬勵的確認在其鑄幣交易裏的「兌奬槽」裏標記。每一個鑄造者需要對前端240個區塊內的奬勵目標進行評估：**如果目標曾經是自己連接並交互良好的公共服務節點，則認可該筆奬勵**（兌奬槽內相應位置1），否則不予承認。每個區塊的外部奬勵為3名，240個區塊共計720個標記位。

如果服務節點因為比較少的確認數未能提取全部奬金，截留部分將由下一個評估段（240塊之後）的對應塊迴收。迴收的金額會作為鑄幣收益的一部分，一起參與到按比例的正常分配。


### 局部評估的合理性

由於兌奬的約束，鑄造者的隨意選擇其實並非眞正的隨意，奬勵給一個並不提供服務的私人地址沒有用，因為其它鑄造者不會知道它，而如果選擇一個自己開設的服務，其實沒啥可說的，因為提供了服務，而且效率和知名度也需要付出。當然，這對僅僅提供公共服務卻沒有參與鑄造的人來說有些不公，這確實是一種遺憾，事無完美。或許可以欣慰的一點是，這種情況可能並不多，而且也並不是所有的鑄造者都會運行公共服務。

提供良好公共服務的節點會被鑄造者記住並緩存，如果鑄造者奬勵自己並不高效的服務器，確認數很可能不高，奬金會被截留。而如果某個專業的公共服務器強大並被很多節點喜歡，它的奬金兌現比例就應該問題不大。通過奬金截留來消解自私行為的收益，或許有些彌補作用吧。

P2P的邏輯是鬆散的，一個節點對週圍節點的感知並不多，如果公共服務的節點很多，應用節點通常是就近連接，所以鑄造者感知的服務節點也不會有太大的交集，因此上面設計的確認數不高。局部範圍是存在的，一個局部範圍內可能存在某個固定的優勢服務器從而形成壟斷，但鑄造者是自由的。

源於鑄造者的隨機性、評估的自由、以及局部範圍本身的動態變化，可以認為全網範圍內的奬勵目標也是隨機的。

> **註：**<br>
> 這種評估受限於鑄造者對服務節點的觀察範圍，而不是在全網範圍內衡量，因此稱之為局部評估。<br>


#### 附：節點評估的緩存池規模

P2P公共服務網絡的規模可能很大，這樣服務節點就會非常地離散，節點需要對提供過服務的節點進行緩存，從而實現足夠的確認行為。

這裏假設服務網絡的規模是10,000個節點，240個區塊按最高5次確認，緩存池的大小估算如下：

- **命中率**：`240/5 = 48` （卽每1個確認需要由48個區塊來命中）。
- **等效集**：`10000/48 ≈ 200` （保持相同的命中率需要的集合大小）。
- **池大小**：設定冗餘度為1.5，緩存池大小為 `200 x 1.5 = 300`。

這是一種很簡單的估算法，在實際運行中我們可以保留評估的節點規模大概為 `400 ~ 600`，然後僅僅緩存自己認可的節點。如果網絡規模變大，或者發現上面的估算不合理（通過抽檢確認數發現），可以調整評估者規模。這是一個可運行時優化的參數。
