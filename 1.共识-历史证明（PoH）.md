区块链是区块的链表，区块包含了不定数量的交易，如果有一笔交易进入了区块，就相当于该交易成了事实，也称为被 *确认*。

区块是交易的集合，任意节点都可以验证交易然后打包创建，但谁打包的区块才算数呢？这就需要共识，一种机制，来决定谁打包的区块可以进入主链，也即：谁拥有区块的铸造权。


## 基于历史的证明（PoH: Proof of Historical）

交易ID是交易数据的哈希摘要，拥有很强的随机性，如果把交易ID用作一个评选因子，是否就可以获得某种随机的筛选呢？

交易是任何一位用户都可以创建的，拥有普适性，同时，交易ID作为历史数据无法更改，它又拥有确定性。与传统的PoS权益证明不同，交易ID也与财富无关，没有富者越富的逻辑。所有的交易ID都是平等的，唯一增加获选机会的方式就是创建更多的交易。

创建交易有交易费成本，但毫无疑问，这种机制会一定程度鼓励交易。更多的交易和交易费可以充实矿工的收益，这是良性的。


### 哈希塑造

采用数据的哈希序列进行评比可能会出现一种攻击：

*通过向哈希源数据中加入随机的无用数据，即可改变哈希计算的结果，使其趋近于某种想要的目标*。此即：「**哈希塑造**」。

为避免这种攻击，需要对参与哈希计算的因子进行约束：不应当有可塑性，或可塑性极低，或成本高昂。


### 参与因子

#### 概念

- **铸凭交易**：有资格参与铸造评选的交易称为铸凭交易。
- **铸凭哈希**：以各种因子结合计算的哈希称为铸凭哈希，它是铸造竞争的直接对比值。
- **评参区块**：铸造竞争的源数据应当是动态的。我们用动态增长的链末端来表达，比如取末端某个负高度的区块的相关信息。
- **UTXO指纹**：UTXO是「Unspent Transaction Output」的缩写，可以用UTXO集的指纹来获得某种动态的确定性。
- **币权总值**：交易的 `币量*币龄` 称为币权。一个区块收录的交易的币权合计（总值），可以作为一个动态的半确定性因子。

> **注：**
> 币金被交易花费后，其新输出的币龄为零。也即「币权销毁」。
> 币龄按交易实际被收录入区块的时间计算，而非通过交易的时间戳计算。


#### 可塑性分析

*交易ID*是交易数据的哈希摘要，用户可以通过在交易里添加冗余数据来任意塑造交易ID。因此交易ID必须位于末端动态参考位置之前的区域，使得在交易创建时，没有信息可供针对，无法塑造。

*区块ID*的逻辑类似，铸造者也可以往Coinbase交易里填充信息（同时还能自由组织交易序列），因此不能采用。

UTXO集的*指纹*是当前区块链所有UTXO的总结，改变该指纹需要实际创建交易并花费，有交易费成本。**注**：UTXO指纹的计算包含了交易ID，因此其可塑性依然强大。

末端区块的*时间戳*是动态的，但区块创建有严格的时间间隔，因此时间戳没有可塑性。**注**：区块时间戳并非实际存在的字段，而是通过创始块时间戳和高度计算得出。

*铸凭哈希*是最终的计算结果，如果设计可行，铸凭哈希是市场化自由竞争的结果，是否可预期（或塑造）取决于算法本身是否合理。每一个区块都有自己的铸造者（也即铸凭哈希），动态的链末端逻辑可以提供随机的确定性。

**币权总值**由区块收录的全部交易决定，如果要改变币权总值，就需要改变区块所收录的交易。添加新的交易会有交易费成本，而移除交易则有交易费损失，因此币权总值的可塑性很低，成本也较高。


### 规则

- 铸凭交易：有效区间为 `[-80000 ~ -240]` 号区块高度。设计截止时间可提供一种淘汰机制。
- 评参区块：取链末端的 `-8` 号区块。这是一个综合权衡的折衷值，考虑了时间长度（7*6=42 分钟）对参与者的友好性。
- 币权销毁：取链末端的 `-32` 号区块的币权总值。与评参区块相距较远，以抑制铸造者塑造币权的动机。
- 铸造者：  交易头中指定的铸凭公钥哈希，或交易里首笔输入的花费者公钥哈希。该账户即为铸造者（用其私钥签名）。

淘汰机制可以让用户有机会抛弃过往（比如委托了不良的第三方），同时也让攻击者总是需要不断地积累新的资源……

排除末端 `-240` 个区块（1天）内的交易，以阻止可能的交易ID塑造。因为交易ID塑造成本极低，所以距离应当足够远：攻击者需要精确预测未来第 `232` 号区块的铸凭哈希和币权销毁总值——这基本上是不可能的。

币权销毁使用 `-32` 号区块的值，用于抑制铸造者对币权的塑造：攻击者需要预测未来第 `24` 号评参区块的铸凭哈希，即便该哈希会提前几个区块出现在择优池中，距离依然较远。辅以高成本，应当能有效抑制对币权的塑造。

> #### 设计注记
> 币权销毁如果使用评参区块的值，会让塑造成为一个正常的可选项，唯一的制约仅为成本。
> 使用 `-32` 号区块而非更远位置的区块，是因为这个距离应当已经足够（如果后期有必要，可以调整）。

> **提示：**
> 一个新区块创建后，它就会作为未来的评参区块被瞄准，竞争者据此评估自己的资格。


### 铸凭哈希

> **说明：**
> 下面的伪代码类似Go语言语法，但仅用于表意。

为抑制对潜在铸造者的实时分析和发掘，采用抗ASIC矿机的 `Equi-X` PoW 哈希算法：

```go
Mix := 0x517cc1b727220a95   // 混合常数，增加Bit位复杂性（FxHash 算法常量）
Height := n                 // 当前待铸区块高度（动态确定性）
X := Bytes(Height * Mix)    // 转为字节序列（无损）

// 铸凭哈希种子：
// 加入铸造者公钥以增强信息熵并绑定身份。
// @Stakes -32号区块币权总值，聪时数（已存在于区块头内）
// @pubKey MintPKHash 或 LeadPKHash 的源公钥数据
var challengeSeed []byte = BLAKE3_256( pubKey || 铸凭交易ID || Stakes || 评参区块:铸凭哈希 || X )

// 客户端：计算铸凭哈希
// 规则：
// - nonce 必须从当前区块高度开始（≥Height）。
// - solution 内的索引值必须有序（升序且无重复，确定性）。
// - hashList 按 solution 内的索引值分别计算出的哈希清单（一一对应）。
hashList, solution, nonce, err := equix.Solve(challengeSeed)
mintHash := BLAKE3_256(domainTag || hashList...)

// 验证者：验证铸凭哈希
// 基本规则检查：优先排除不合法项。
if nonce < Height || solution 非升序且无重复 {
    return false
}
// hashList 哈希成员，与 solution 内的索引值一一对应。
// isValid 哈希工作量是否有效。
hashList, isValid := equix.Verify(challengeSeed, nonce, solution)
if !isValid {
    return false
}
mintHash := BLAKE3_256(domainTag || hashList...)
```

**评选**：
1. 先按 nonce 分组，值越低优先级越高。避免尝试不同的 nonce 引发算力竞争。
2. 在 nonce 相同的条件下，铸凭哈希值越小权重越大。

> #### 设计注记
> 实际上无法用*签名数据*计算铸凭哈希来隐藏信息，因为签名数据通常有可塑性：
> 只要算法中有自由的随机数，就可以通过定制算法创建可塑性，这会带来类似 PoW 的竞争，失去公平性。
>
> Equi-X 在同一 challengeSeed 和 nonce 时也可能存在多个解，但问题不大。

参与铸凭哈希计算的因子基本都是公开的，没有隐蔽性。这有一个好处：外部观察者可以分析（抽查）铸造竞争的参与状况。但也因此，攻击者可以分析发掘未参与铸造竞争的高权重者，从而找到并收买贿赂……

采用抗 ASIC 的 PoW 哈希算法，可以很大程度上抑制这种实时分析发掘。如果要进行分叉攻击，攻击者最多可收买8个更高权重者（分叉超8个区块后，评参区块就不同了），但只有20个区块（2小时）的时间窗口，攻击的难度和成本会比较高。

计算铸凭哈希需要完成工作量，这对大平台或钱包运营商来说不友好。这可能促使它们更愿意让用户自己参与竞争，而这对去中心化来说，应该是件好事。


### 初段规则

区块链最初创建的8个区块没有 `-8` 号评参区块，因此需要一些特别的处理。

涉及两个基本变量：

1. 评参区块高度获取。
2. 交易所在区块的高度的合法性。


#### 评参区块

因为铸造竞争源于交易ID，初期的交易只由区块链的创建者发布，没有攻击者，因此可以采用简单的算法如下：

```go
// 初段：
// 评参区块取创始块即可
// @currentHeight 当前区块高度
if currentHeight < 8 {
    return 0
}
// 后期：正常的取值。
return currentHeight - 8
```


#### 铸凭交易

源于同上没有攻击者的原因，区块创建者不需要塑造交易ID，因此简单地判断即可：

```go
// 初段：
// 交易所在区块高度是否合法。
// 放宽一天的交易量，避免进入正常判断后仅有第240号区块的交易有效。
// @currentHeight 当前区块高度
// @txHeight 交易所在区块高度
if currentHeight < 480 {
    return txHeight < currentHeight
}
// 后期：正常的判断
h := currentHeight - txHeight
return h > 239 && h <= 80000
```

任何合法的交易ID皆可作为铸凭交易参与竞争。第240块之前的参与因子可灵活处理。



## 铸造者的预选

因为评参区块是链末端 `-8号` 区块，所以一个铸造候选者最多可提前 `7个` 区块时段得知要对比的目标。如果一个节点及时评估并广播自己的铸凭交易（及其凭证），则全网会有充足的时间进行预先沟通。尽量多的参与者加入，可以使尽可能优质的铸凭交易被发掘出来。


### 择优池

在铸造候选者的预先沟通中，各个节点会收集广播出来的铸凭交易，按品质排序放入一个缓存池，这个缓存池就是 **择优池**。择优池实际上是一个池集，因为每一个评参区块都对应一个择优池。

新的铸凭交易加入择优池的逻辑很简单：计算其铸凭哈希，如果比池中最差的一个好，则有序插入并转播，否则忽略。


### 择优凭证

铸造候选者的证明包含如下几项信息：

- 高度：交易实际所在区块高度。
- 交易ID：用于验证并参与铸凭哈希计算。
- 工作量解答：Equi-X 的解答数据（`solution` 和 `nonce`）。
- 铸造者公钥：用于验证铸造者身份和签名正确性。
- 铸凭哈希：即 `mintHash` 值本身，简单携带（可从上面相关信息推导计算出来），便于检索。
- 签名数据：铸造者对 `mintHash` 的签名。

> **注意：**
> 最终的铸造者的*择优凭证*会存储在区块的 `Coinbase` 交易中，作为交易数据的一部分参与交易ID的计算。
> 凭证中包含的签名是铸造资格证明材料，不是交易输入项花费签名或证明拥有交易的签名，不属于后期剪枝的范畴。


### 择优池同步

为避免刚上线的优质竞争者带来扰乱（引起分叉），动态更新的择优池需要在各节点间提前确定下来，这就是择优池的同步。

一个新区块在成为评参区块（`-8号`）之前，铸造竞争有如下分段逻辑：

1. **广播收集**：新区块创建后，到它成为 `-6号` 区块前，有**5**个区块时段可以对铸凭交易计算铸凭哈希，广播并优选。
2. **同步优化**：当该区块成为 `-6号` 区块后，择优池的更新结束，进入同步期。耗时**2**个区块时段。
3. **抵达结束**：当该区块成为 `-8号` 区块后，即成为当前待创建区块的*评参区块*。此时其择优池中的候选者已确定。


#### 广播收集

除了接收铸凭交易的广播和评估外，此阶段节点还会彼此沟通择优池状况，以期最优化择优池成员。

广播收集历经5个区块时段，时间裕度30分钟。较长的时间使得在进入同步阶段后，大部分节点的择优池（区块特定）状况应该差异不大。


#### 授权节点约束

择优池大小设计为 `20`，有权同步的节点为择优池中 `后15` 名成员。这基于利益无关性考虑：

*在P2P环境下，时间无法准确约束，如果没有限制，优质的迟到竞争者可以随时把自己加入择优池。这会导致同步工作难以正常结束。*

有了此授权节点约束，如果优质的候选者迟到了，排名靠后的节点没有动力去把它额外补入。而一个中等优质的迟到者也没有动力把自己加到择优池，然后开启新的同步，因为它基本不可能成为铸造者。

> **注：**
> 一个授权节点只有一次同步的权力。


#### 同步流程

**约定**：每个同步者提供的择优池称为*目标池*，接收者自己的择优池称为*本地池*。

- **裁判池**：创建一个本地池的副本，合并目标池，即为裁判池。判断对端是否在裁判池里的后段15位中，如果不在，则拒绝同步。否则缓存目标池（单独形式暂存）。
- **预合并**：同步时间截止后，合并所有暂存的目标池，取合法范围（后段15位）内的成员，即为授权节点集。
- **终合并**：对授权节点集成员的目标池进行合并，即得到最终的择优池。

对于一个新上线的节点，其本地池为空，但这不影响裁判池的创建（此时裁判池就是对端的目标池）。

这是一种P2P环境下的概略性同步，不是拥有唯一性的精确约束。但设计通过足够长的时间宽容度来尽可能优化结果，不过效果有待验证。

如果最终的择优池没有唯一性，会导致分叉。这由后续的分叉竞争机制来解决，以最终收敛到一个唯一的主链。



## 附：铸造者验证

> **前提：**
> 区块头链是验证的终点，每个客户端都持有区块头链。

铸造竞争者在广播自己的铸造资格时，除了*择优凭证*外，还需要提供如下信息：

- 交易ID到哈希树根（`TreeRoot`）的默克尔验证路径（叶子节点含3字节前置序号）。
- 如果铸凭交易未设置 `MintPKHash`，则需要提供 `ListHash` 以便于计算输入项根哈希。


### 信息检索

验证者利用择优凭证中的高度和交易ID信息，检索需要的数据：

1. 交易头：输入段根哈希。
2. 交易头：输出段根哈希。
3. 交易头：交易时间戳。
4. 交易头：铸凭公钥哈希（`MintPKHash`），可能为空。
5. 目标区块的 UTXO/UTCO 集合指纹值。

> **提示：**<br>
> 当高度在真实年初1天内时，如果当年度检索无果，可年度减一再检索一次。**注**：外部服务使用真实年度。<br>
> 因为区块不收录未来的交易，所以仅需考虑年初回退的情况。**注**：检索年度是按交易时间戳计算的。<br>

> **实现：**
> 作为一种优化，检索的信息也可以由铸造竞争者自己提供。


### 验证：交易ID是否合法

根据*择优凭证*中的信息和上面的检索结果：

1. 从区块高度验证是否在合法范围内（`[-80000, -240]`）。
2. 从铸造者公钥计算公钥哈希（可能为 `MintPKHash` 或 `LeadPKHash`）。
3. 根据检索结果中的 `MintPKHash` 情况：
    - 非空：则从公钥计算的公钥哈希应当与检索结果相同。
    - 为空：计算的公钥哈希作为 `LeadPKHash` 参与输入段根哈希计算：Hash( `ListHash || LeadPKHash` )。
4. 如果计算的公钥哈希被判定为 `LeadPKHash`，则可先验证输入段根哈希是否匹配。
5. 结合*检索结果*中的 `1-4` 信息计算交易头哈希，验证是否与*择优凭证*中的交易ID匹配。


### 验证：交易ID是否属于目标区块

- 从区块高度检索区块头链中的区块头信息，提取其中的 `CheckRoot` 值。
- 结合*检索结果*中的第 `5` 项信息和用户提交的默克尔验证路径，核实交易ID属于该区块（`CheckRoot` 匹配）。


### 验证：铸造者身份

- 计算实际的 `mintHash` 值，对比*择优凭证*中携带的 `mintHash` 是否匹配。
- 验证*择优凭证*中的对 `mintHash` 的签名是否有效。



## 设计参数

- 铸凭交易的区块有效范围：`[-80000, -240]`（约11个月）。
- 评参区块为 `-8号` 区块，取其铸凭哈希参与当前区块的铸凭哈希计算。
- 币权销毁因子采用 `-32号` 区块的币权总值，抑制铸造者对塑造币权的动机。
- 择优池容量为 `20名` 候选者，内部按铸凭哈希字节序列从小到大排列（值小者优）。
- 择优池中排序在前5名之后的 `15位` 候选者有权发起择优池同步。
