# Blockchain Core 核心实现

> **主旨**：极致简化，专注核心。方便任意场景下的自由集成。


## 需求

Blockchain Core 是区块链系统的核心组件，主要着眼于区块链整体的管理与维护。

核心应当精简并独立运行，可以与任意其它功能的部件协作，共同完成区块链生态的自由构建……比如被集成进普通用户使用的轻客户端、简单参与铸造竞争的围观者、正式参与交易校验的校验组各成员、公共服务器、浏览器插件、移动应用等等。

因为这种极简，共识构造也应当被剥离出来（只有校验节点才需要）。作为普通的用户轻客户端，公共服务的第三方节点等，它们只是区块链的用户，会默认区块链是安全的。


## 区块头

区块的ID由区块头计算而来，连续的区块头就形成了区块头链。


### 结构

区块头结构（伪代码）示意：

```go
BlockHeader: {
    Version   uint32    // 版本号
    Height    uint32    // 区块高度，代替时间戳信息
    PrevBlock [48]byte  // 前一区块哈希
    CheckRoot [48]byte  // 校验根（由交易哈希树根、UTXO/UTCO指纹合并计算）
    Stakes    uint64    // 币权销毁，单位：聪时（聪*小时）
    YearBlock [48]byte  // 前一年块哈希，可选（仅 height % 87661 == 0 时存在）
}
```

**其中：**
- 币权销毁可表达交易量，是新区块选择的一个竞争因子。另外也作为铸凭哈希计算因子。
- 年块指从创始区块开始，每年引用一次上一个年块（按87661/年）。可用于更高效和节省地引用主链，仅在年块高度存在。
- 没有时间戳字段，时间戳通过高度和时间间隔计算出来（创始区块时间戳已知）。

如果整数按4字节计算，区块头长112字节（年块除外，平时无需包含），区块头年数据量：112x87661 ~= **9.36**MB。

> **币权：**
> 未花费输出项金额和存留时间的乘积，一旦花费则币权归零。
> 时间单位为小时，不足1小时即为零。可滤除高频交易影响（某些攻击）。


### 交易约束

区块头内包含的 CheckRoot 由区块内所有交易的哈希树根和前一区块完成后的 UTXO/UTCO 集的指纹合并计算而来。

交易的哈希树采用 Merkle 树逻辑，但每一层级的节点排布用数组结构，以支持更高效的单笔交易验证和位置检索，叶子节点包含前置3字节序号，用于未来的某些功能。

## 功能

### 入块验证

接受外部组件的区块提交，简单验证区块的合法性即可。
如果有相同高度的区块二次提交，简单反馈冲突错误（拒绝入块），外部需删除该区块后才能重新提交入链。

> **注：**
> 简单验证即可，比如新区块头的前一区块哈希值是否正确。
> 区块本身的合法性并不由核心负责，而是由功能剥离的外部提交者负责。


### 数据存储

根据用户配置，存储区块头链数据。区块头链需从创世块开始维持连续性，直至当前最新区块不可中断。

因为区块头支持年块哈希，所以实际上并不需要存储完整的区块头数据，只要年块哈希衔接，就可以维持链的连续性。用户可以配置任意年度的区块头数据忽略，只需维持年块的衔接即可。


### 数据完整性

由于核心不存储完整的区块头数据（年块衔接机制下的省略），但同时却提供完整的区块头链检索。因此需要一个确保完整性的机制：就是可以连接到第三方公共服务（`Blockqs`），以即时获取完整的区块头数据。

可能需要从多个第三方服务节点获取数据，以确保数据的可用性与完整性。


### 数据访问

- 提供区块头数据的访问接口，支持按区块高度、区块哈希值查询区块头。
- 支持区块头链的同步接口，方便外部组件调用进行区块头链的同步。
- 支持按年度查询区块头数据，方便快速定位特定年份的区块头。
- ……（待定）


### 手动切换主链

如果全球网络出现了分区超过2小时，20+个区块的分叉长度就无法参与竞争了。此时两条链都会被认为是合法的，客户端应当允许用户手动选择自己认可的主链。

这可能很简单：只需要用户手动选择目标分叉的几个主要节点，然后请求分叉之后的区块头链数据即可。

> **附注：**
> 人工切换主链类似于用脚投票，这不是算法的逻辑，而是社会性的竞争与选择。
> 这在未来天基互联网强连接的环境下，应当很难发生。



## 哈希策略

在本系统设计中，除了由签名保证交易入块时的即时安全性外，区块链的长期历史安全完全由哈希约束来保证。

- **区块头**：`SHA3-384`。兼顾量子安全与数据量。
- **CheckRoot**: `SHA3-384`。Hash384(`TreeRoot || UTXORoot || UTCORoot`)，一次性开销，成本可控。
- **树结构**：`BLAKE3:256`。处理多哈希场景，提高性能并节省内存。
- **树叶子节点**：`SHA3-384`。默认情况下，树结构的末端叶子节点采用与树枝段不同的算法（除非另有说明）。
- **交易头**：`SHA3-384`。与区块头设计一致。
- **附件指纹**：`SHA3-512`。确保超长期安全，与附件片组哈希可形成双哈希耦合保护。
- **公钥哈希**：分单公钥哈希和复合公钥哈希，算法：`SHA3-256(BLAKE2b-512)`，详见 [附.交易](附.交易.md)。
- **未花费/未转出数据集**：`SHA3-384`。UTXO/UTCO 末端叶子节点数据摘要。
- **铸凭哈希**：`BLAKE3-256`。可能有大量参与者，关注速度与效率。

> **注意：**
> UTXORoot/UTCORoot 针对的是前一区块完成后的 UTXO/UTCO 集，这样当前区块才有充裕的时间完成计算。
> 同时这也是 UTXO/UTCO 集指纹链式约束的关键，一种重要的安全机制。


### 哈希树结构

- UTXO 集指纹，4层宽成员哈希校验树。
- UTCO 集指纹，结构同上。
- 区块交易集的二元哈希校验树（含序，类 Merkle）。
- 交易输入集哈希树，仅2层：叶子 + 根。
- 交易输出集哈希树：`RootHash = Hash256( Tree<Outputs> )`。
- 附件片组集哈希树（分片数据哈希校验树，含序，类 Merkle）。


### 附：公钥哈希的量子安全强度

公钥哈希（账户地址）采用 `SHA3-256(BLAKE2b-512)`，输出 32 字节，提供约 128 位量子安全强度（SHA3-256 的 Grover 攻击下限）。

系统中其他大多数哈希使用 SHA3-384（192 位量子安全强度）。账户地址的生命周期与 UTXO 绑定，未被花费的地址会持续暴露在链上。这是未来量子计算机成熟后，系统中最先受到压力的哈希环节。

此设计是空间节省（32B vs 48B）与安全性之间的有意权衡，辅以抗量子签名算法，风险可接受。



## 区块链启动

### 创世区块

创世区块是区块链的起点，锚定所有后续验证逻辑。它是每一个客户端都需要硬编码包含的数据。


#### 区块头

- `Version`：版本号固定为 `1`。
- `Height`：高度固定为 `0`。
- `PrevBlock`：创世区块没有前一区块，此值为全零。
- `CheckRoot`：区块仅包含一笔 `Coinbase` 交易，关联的 UTXO/UTCO 集指纹为全零（但正常串连计算哈希）。
- `Stakes`：无币权销毁，此值为 `0`。
- `YearBlock`：没有前一年块，但 `0 % 87661 == 0`，因此 `YearBlock` 字段存在，但值为全零。

> **注：**
> 交易哈希树依然遵循常规，即使只有一笔交易，叶子节点也前置3字节序号（`000`）计算哈希树根。


#### 创世交易（Coinbase）

**交易头数据：**

```go
TxHeader: {
    Version     uint16      // 版本号：1
    HashInputs  Hash256     // Coinbase 交易没有输入项，此字段省略
    HashOutputs Hash256     // 正常计算：BLAKE3-256(SHA3-384(LeafData))
    Timestamp   int64       // 创世时间戳（毫秒）。也即区块的时间戳，后续区块时间戳由此推导。
    MintPKHash  [32]byte    // Coinbase 交易没有首领输入，因此定义铸凭公钥哈希。

    // 以下为Coinbase交易特有的字段：
    BlockHeight uint32      // 交易所在区块高度，创世区块为0
    Minter      *MintProof  // 择优凭证：创世块没有该值，省略
    FreeData    []byte      // 创世块声明（<256字节），可能包含一些有意义的句子
    BurnCoin    int64       // 创世块没有交易费，此字段为0
}
```

**币金输出项：**

- 接收者：币金的接收者公钥哈希，共2名（铸凭者 + 校验组）。
- 金额：`10_000_000_00 chx`（即 10 币），按 40:10 比例分配：
    - 铸凭者：`2_000_000_00 chx`（2 币）。
    - 校验组：`8_000_000_00 chx`（8 币）。
- 锁定脚本：标准的锁定验证逻辑：`SYS_CHKPASS`，简单地址核实&签名验证。

> **注：**
> 百日内无公共服务奖励（见下文），因此省略公共服务的输出项。

> #### 关于见证信息
> 对 Coinbase 交易的签名是对整个交易（TxID）进行签名，没有授权种类的区分。
> 依循常规，该签名数据（见证信息）依然是独立的，可剪枝。
>
> 但作为创世区块，铸造者对区块 `CheckRoot` 的签名需要保留，这是区块链根的锚定。


#### 启动

创世交易是一笔 Coinbase 交易，由客户端构建。作为一个特例，`MintPKHash` 即为铸造者，它同时会对交易和区块（`CheckRoot`）进行签名。

因为没有竞争者，下一个区块（#1）也将由创世区块的 `MintPKHash` 继续铸造。但 #1 号区块中可以包含2笔普通的花费交易，输入项即为创世区块的新币。

到 #2 号区块时，区块铸造已经有竞争者了（详见 [共识-历史证明#铸凭交易](1.共识-历史证明（PoH）.md#铸凭交易)，初段不受 `-240` 区块范围限制）。至此，从 #1 号区块开始，区块链进入前期运行状态。


### 百日扩张

基于交易历史的证明要求参与者已经是用户，这就需要把铸造的币金分发出去。

除了公共服务可以从外部获取币金之外，对于初期的区块链成长，这里设计了一个百日扩张的运行策略。

- `0号` 区块：*创始区块*。区块链启动。
- `1 - 10号` 区块：*私钥扩张*。创建适量的收益地址，用于之后的交易创建和铸凭交易竞争。
- `11 - 7200号` 区块（至30日）：*币金积累*。观察情况，保留随时停止区块链运行的可能性。
- `7201 - 24000号` 区块（至百日）：*抽奖扩张*。由社区推广活动收集地址，由积累的币金发放奖励，扩展初期铸凭交易规模。

至此，铸币交易（Coinbase）中没有对公共服务的奖励。

- `101日 ~`：接受公共服务，启动对外奖励，Coinbase 交易中必须包含5笔输出（铸凭者、校验组、Blockqs、Depots、STUN）。至此，区块链网络进入正常的自由市场运行……

> **社区共约：**
> 此期间每交易的输出项数量不应超过输入数量的**2**倍，以约束扩张速度。
> 客户端实现可能会拒绝违反该共约的交易。**注**：仅为共约，区块收录它们并不非法。



## 其它

### 初始主链验证

没有任何数据的初始节点上线，需要获取主链信息以及必要的数据以验证之。

主链信息由它连接的公共服务（`Blockqs`）或某个校验组提供。验证信息则包括：

1. 创世区块及其区块头：由客户端自带（硬编码内置）。
2. 区块头链：从当前区块高度到创始块的区块头链，可局部用年块衔接（轻量负载）。
3. 末端部分区块的 Coinbase 交易及其哈希验证路径（含 UTXO/UTCO 指纹）以及铸造者对 `CheckRoot` 的签名。
4. 当前区块高度的 UTXO/UTCO 集和末端部分区块的数据（可选，用于逆推式严格验证）。

> **注：**
> 末端部分区块的长度可能为31个区块，以涵盖分叉竞争区间。
> 铸造者对区块 `CheckRoot` 的签名数据通常会被保留**1年**，末端31个区块则是必须的。

初始上线节点可以向不同的公共服务（`Blockqs`）节点或校验组请求这些数据，相互比对/验证以确保找到了真实的主链。


### 主链和分叉标识

标识一条区块链有如下几项信息：

- Protocol-ID：区分本区块链与其它区块链，值：`Evidcoin@v1` 字符串。
- Chain-ID：当前区块链的运行态：`mainnet`（正式主链）、`testnet`（测试链）、`devnet`（开发链）。
- Genesis-ID：创始块信息，即区块ID。
- Bound-ID：主链绑定。绑定分叉之后的首块ID：`<Height><BlockID[:20]>`，可选。
  - 主要用于分叉后绑定主链，避免新交易在支链上被重放。
  - 随着时间推移，分叉成为另一条正常链后，其 `Protocol-ID` 应当变化。此后即不再必须。

该标识信息参与交易的签名，但仅作为链区分放在交易消息的前段：

```go
// 识别信息前置
// TxMSG 为交易相关的签名消息（有内在结构）
MixData = ( Protocol-ID || Chain-ID || Genesis-ID || Bound-ID ) || TxMSG
signData = Sign( MixData )
```

后期签名数据将会被剪枝，区块链的长期安全由哈希约束保证。

> **提示：**
> 该标识信息也用于节点在 P2P 连接握手时声明身份，以识别彼此。


### 区块限额

区块限定仅为对数据量（尺寸）的约束，数据包含解锁脚本但不含见证信息（定制验证放入解锁脚本的签名数据计入）：

- 第 `1~3` 月：7305*3 = 21915 块，固定为 **1MB**。
- 第 `4~12` 月：按每月7305块计算，逐月递增 **1MB**，最终到 **10**MB。**注**：最后一月为7306个区块。
- 第 `2` 年起：按恒星年（87661/年）计算，逐年递增 **1MB**。

> **注：**
> 年度按恒星年 `87661` 个区块计算，则每月为 87661/12 ~= *7305* 个区块。
> 最后一月容纳余出的尾数（1个区块），以实现完整的一年（7305*11 + 7306 = 87661）。
