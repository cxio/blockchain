# 交易

不同于Bitcoin的交易历史只是一种存储，这里的交易历史有后期的各种需求（识别脚本应用、介管脚本引用、第三方二次开发等）。因此交易的数据结构需要便于检索和验证。

检索和验证主要采用哈希校验树的方式，类似于Bitcoin的默克尔树，但细节稍有不同。


## 交易头

类似于区块头抽象了区块数据，这里设计了交易头结构。伪代码示意：

```go
// 简化版示意：
TxHeader {
    Version     uint16      // 版本
    HashInputs  Hash256     // 输入项根哈希（BLAKE3:256），Coinbase 省略该字段
    HashOutputs Hash256     // 输出项根哈希（BLAKE3:256），即输出项集的哈希树根
    Timestamp   int64       // 交易时间戳（Unix 毫秒）
    MintPKHash  [32]byte    // 铸凭公钥哈希，可选（空值时省略），Coinbase交易通常会设置此值

    // 以下为 Coinbase 交易特有的字段（普通交易省略）：
    BlockHeight uint32      // 交易所在区块高度
    Minter      *MintProof  // 铸造者：择优凭证
    FreeData    []byte      // 自由数据
    BurnCoin    int64       // 销毁的交易费（50%）
    AwardSlots  [18]byte    // 公共服务兑奖槽标记位
}

// 交易头的哈希即为交易ID。
// 算法：SHA3-384
TxID: Hash384( TxHeader )
```


#### 关于铸凭公钥哈希（`MintPKHash`）

公钥哈希是铸造者的身份，在参与铸造竞争时提供相关证明。逻辑上，这与交易的花费无关。

`MintPKHash` 是一个可选项，如果首领输入源公钥哈希是多签地址，就可以在这里设置一个简单的单签公钥哈希（对于铸造竞争场景，多签的困难度没有意义）。同时，如果用户想单独定义铸造竞争的身份，也可以由此实现。

另外，它实际上也可以解耦签名算法，如果有必要，系统可以在未来基于此更容易地实现。

> **附注：**
> 这还能带来一个小小的优点：*隐藏参与铸凭哈希计算的公钥*。
> 不过这只是一次性的，如果用户参与了择优凭证的广播，公钥就公开了。



## 交易体

由输入项集合和输出项集合构成。下面分布阐述输入项和输出项。


### 输入项

不支持未确认交易的输出项作为输入项。

也即：输入项只能引用已确认交易的输出项（从 UTXO/UTCO 取值），这也是组队校验对交易独立性的要求。


#### 结构

输入项是交易信用的来源，由3个字段构成：*年度*、*交易ID*、*输出序位*。

- `[n]`  交易所在真实年度（按交易时间戳计算），变长整数。这是一个概略的时间分级，便于管理。
- `[16+]` 交易ID的前段局部引用，可变长度。短引用以节省空间。
- `[n]`  输出项下标序位（来源输出集），变长整数。大多数都只需要1字节表达（<128）。

所有的币金输入项都从当前**UTXO**集里检索，凭信输入项从**UTCO**集里检索。UTXO/UTCO 的可靠性由其指纹保证，因此可变长字节序列的引用只涉及碰撞问题。

短引用由用户自由设定（**≥16 Bytes**），如果在 UTXO/UTCO 集里存在碰撞（单年度内），规则为：

*末端叶子节点按交易ID排序，首个匹配即引用。不区分是否碰撞*。

这样，如果引用错误，交易将无法验证通过。用户应当预先查询 UTXO/UTCO 集，确认自己交易中的输入引用没有碰撞，否则应延长引用ID的字节数。这是用户自己的责任。

> #### 设计注记：
> 短引用不能设计为固定长度（如20字节），这可能导致一种攻击：
> - 攻击者创建一笔交易，使其交易ID前20字节与某笔未花费输出的交易ID前20字节相同。
> - 因为短引用出现了碰撞，那笔被攻击的未花费输出即可能无法被正确引用（无法被花费），攻击成功。
>
> 这在未来哈希算力提升后，攻击成功的可能性变大。但允许变长，则无此问题。


#### 哈希校验树

交易里的输入集被独立计算哈希树根，因为对输入项的后期检索&验证无需求，哈希树设计为2层：叶子 + 根。

```go
// 输入项根哈希：
// @ListHash 输入项串连哈希
// @LeadPKHash 首领输入源公钥哈希，参与哈希计算以便于铸造者验证。
// BLAKE3-256
InputHash: Hash256(ListHash + LeadPKHash)

// 输入项作为列表串接计算哈希。
// SHA3-384
ListHash: Hash384( ListOf(Inputs) )

// 不含见证信息
Input: {
    Year         int      // 交易年度
    TxIDPart     []byte   // 交易ID前16+字节
    OutIndex     int      // 输出项序位
    UnlockScript []byte   // 解锁脚本
}
```


#### 合法性验证

一笔交易中可以包含多个输入项，每个输入项是另一笔交易未花费/转移的输出（UTXO/UTCO）。输入项需要被验证，以证明交易的构造者拥有它。

验证采用解锁数据（见证信息 + 解锁脚本）的方式：

1. *见证信息* => 进入系统环境，准备验证数据。
2. *锁定脚本：SYS_CHKPASS* => 触发系统内置验证，验证数据由环境提供。**注**：可选。没有则为定制验证。
3. *解锁脚本 + 锁定脚本* => 如果整个脚本执行成功，则验证通过*。

> **注：**
> 锁定脚本中的 `SYS_CHKPASS` 可处于任意位置，但通常设置在脚本的开头。


#### 附：UTXO/UTCO 集合

所有已确认交易的未花费币金构成了 UTXO 集，所有已确认交易的未转移凭信输出构成了 UTCO 集，它们确定于上一个区块。

在本设计中，一个未确认交易的输出不属于 UTXO/UTCO，即便它合法，因为未确认的交易可能被替换或过期失效。同时，这也是组队校验对交易独立性的要求。

试图构建一个*链式的*未确认支付序列是不可能的，只有头部第一笔交易会被验证和广播，其余被丢弃。


### 输出项

接受币金支付或凭信转移的目标（目的地）。

一笔交易中可以有多个输出项。对于币金交易来说，可能包含一个收款者和一个找零地址。对于凭信转移交易，则逻辑上只有一个接收者。

每一笔*输出*包含了特定类型「信元」的相关字段信息（详见『信用结构』中相应的信元定义）。


#### 配置

输出项包含**1**字节的配置。高4位为位标记，低4位为类型值。

```go
 7] 账户摘要：指示接收者使用哈希摘要参与计算输出项哈希，通常无用。
 6] 内容摘要：指示内容部分使用哈希摘要参与计算输出项哈希，主要用于大负载。
 5] 脚本摘要：指示脚本部分使用哈希摘要参与计算输出项哈希，主要用于长脚本。
 4] （未用）
[4] 类型说明：低4位值。
    0) 预留：暂无意义。
    1) 币金：输出为币金支付。可以作为输入源。
    2) 凭信：输出为凭信。可以作为输入源。
    3) 存证：输出为存证，不能作为输入源。常用于介管脚本定义。
```

> **提示：**
> 负载/脚本摘要主要用于单项（负载/脚本）验证时，
> 比如执行 `GOTO` 跳转到目标脚本，用户仅需提供负载哈希和脚本，即可验证。

这里的**内容**指输出项的内容部分，即除脚本和接收者之外的全部条目。**摘要**使用与叶节点相同的哈希算法（`SHA3-384`）。

摘要标记是通用的，适用于全部3种信元类型。不过在实际应用中，内容摘要主要用于凭信和存证，因为它们被引用的可能性更大。

> **注记：**
> 该部分的字段划分与签名授权的种类相同，但两者之间没有关系。
> 也即签名消息不查看该配置（会针对实际的内容）。


#### 哈希校验树

各个输出项作为叶子节点计算哈希，按二元树结构组织成哈希校验树。哈希树根与输入项哈希一起，合并计算交易体哈希。

```go
// 树干 BLAKE3-256，叶子节点 SHA3-384
OutputHash: Hash256:Tree<Outputs>

// 输出项结构示意
// 详见『信用结构』中相应的信元定义。
Output: {
    Serial      int         // 输出序位（从0开始）
    Address     [32]byte    // 接收地址（公钥哈希）
    Amount      int64       // 币金数量（按基本单位）
    LockScript  []byte      // 锁定脚本
    ...                     // 其它可能的字段
}
```

> **注：**
> 这样的二元哈希校验树便于独立验证和定位。


#### 场景：介管脚本

脚本允许在交易间跳转（`GOTO/EMBED`），因此这创建了一个*中介*的逻辑。

接收跳转的目标交易的输出脚本通常被称为介管脚本，也通常是采用存证类型的输出项。它的存在使得交易之间可以形成一个*链式的*关系，便于构建复杂的业务逻辑。

介管的存在，也使得脚本中可以嵌入监管的逻辑，比如一笔先期建立的财务监听脚本，可以接收企业支付脚本的*途经*路过……获取必要的信息，然后……

通常来说，介管脚本的构造者会设置输出项配置中的内容摘要位（如果有较长内容的话），以方便第三方引用时可通过提供内容哈希（而非内容本身）来实现轻量验证。



## 签名

对输出项的*拥有权*通常通过签名来证明。这也是系统内置验证的标准方式。

签名可以是简单的**单签名**模式，即输入项的来源输出是属于一个单公钥哈希，因此也仅由一个私钥对应。

签名还可以是复杂的**多重签名**，即输入项的来源输出是属于一个复合公钥哈希地址，该地址实际上由多个公钥构成，因此也就对应多个私钥。

- **单公钥哈希**：由一个私钥对应一个公钥，对公钥的哈希结果。
- **复合公钥哈希**：由多个公钥根据规则构造的序列，对该序列的哈希结果。因此对应多个私钥。

**注：**
- 单公钥哈希与复合公钥哈希在外观上没有区别（隐私安全），但它们的验证逻辑不同。


### 单签名

交易的输入项来源于单公钥哈希，签名仅需单一私钥。

这是大多数交易的实际情况，即便交易里有多笔输入，它们都来源于单公钥哈希地址。

> **注：**
> 包含多笔单公钥哈希输入的交易，需要这些输入项的全部签名。

单公钥哈希的算法：`SHA3-256( BLAKE2b-512(pubKey) )`。


### 多重签名

适用于复合公钥哈希地址，需要由多个私钥共同进行签名才行。此即为*多重签名*，也即所谓的**多签**（简称）。

多签支持 `M-Of-N` 的工作机制，以增强社会化的安全性和灵活性。

- N：总共需要的私钥数量（参与方的公钥总数）。
- M：实际需要签名授权的最少私钥数量（M ≤ N）。如：`2-of-3` 多签地址表示有3个私钥，任何2个私钥签名即可。

> **注：**
> 内置验证对单签名或多重签名的判断由解锁数据中的见证信息指明。


#### 多重签名的地址

准备 N 个公钥，及 M/N 的配比（如：`2/3`）。

构造流程：

0. 对N个公钥执行简单的哈希运算 `BaseH = BLAKE3-256( pubKey )`，得到N个公钥的*初级哈希*。
1. 对N个公钥的初级哈希按字典序排序&串连：`BaseHAll = Concat( Sort(BaseH1, BaseH2, ... , BaseHn) )`。
2. 在串联结果 `BaseHAll` 前置 `m/N` 配比，计算复合公钥哈希：`PKHmix = SHA3-256( BLAKE2b-512( <m/N> + BaseHAll ) )`。
3. 复合公钥哈希（`PKHmix`）编码为文本形式，即为多重签名的账户地址。

> **说明：**
> `m/N` 配比为 `1+1` 两个字节，因此最多支持 `255/255` 的组合规模。
> `m` 和 `N` 任一值不得为零，否则不合法。

每一位参与者的公钥有一层初级的哈希封装（`BLAKE3-256`），这会更安全。在外观上，多签地址与普通地址也没有区别（隐私安全）。

另外，这种安全性可能让参与者更自由，比如匿名参与会更容易。


#### 多重签名的验证

见证信息：

- **签名集**：`m/N` 中 `m` 个私钥对交易的签名。
- **公钥集**：上面 `m` 个签名对应的公钥序列。
- **补全集**：未参与签名的公钥的初级哈希（`N-m` 个）。

> **注：**
> 组合配比 `m/N` 可由公钥集大小和补全集大小计算而来。

验证过程：

0. 计算 `m/N` 配比。
1. 计算公钥集里各个公钥的初级哈希。
2. 结果集与补全集里的初级哈希混合排序、串联，前置 `m/N` 配比，计算复合公钥哈希。
3. 对比接收者的公钥哈希与上面计算的结果，相同时有效。
4. 验证签名集里的各个签名。


### 签名消息

通过对签名消息的不同约束，签名授权可以有多种模式。


#### 授权种类

由**1**个字节配置签名消息的约束目标：

```go
// 独项：
7] SIGIN_ALL    全部输入项，含解锁脚本。
6] SIGIN_SELF   当前输入项（自身），含解锁脚本。
// 主项：
5] SIGOUT_ALL   全部输出项：适用于币金、凭信、存证
4] SIGOUT_SELF  与当前输入项同序位的输出项
// 辅项：
// 需配合 SIGOUT_SELF | SIGOUT_ALL 使用。
3] SCRIPT       输出项的脚本。
2] CONTENT      输出项的内容：除脚本和接收者之外的全部条目。
1] RECEIVER     输出的接收者。
0] OUTPUT       输出项的全部（脚本+内容+接收者），友好。
```

**说明：**

- 「独项」逻辑自完整。可独立设置，也可与其它设置合并使用。
- 「主项」不能独立设置，需与辅项配合。如：`SIGOUT_ALL|RECEIVER`，锁定全部输出项的接收者。
- 「辅项」需与主项配合使用。如：`SIGOUT_SELF|SCRIPT|CONTENT`，锁定当前输入项同序位输出项的脚本和内容。

> **提示：**<br>
> 所有签名消息都默认包含交易头内的版本信息、时间戳和铸凭公钥哈希（如果有）。<br>
> 未指定的部分不会包含，因此可以被第三方修改而不影响签名验证。<br>
> OUTPUT 选项与 SCRIPT/CONTENT/RECEIVER 为互斥关系，不应同时使用。<br>

**注意**：此授权种类功能仅用于普通交易，Coinbase 交易无此功能，是直接对整个交易进行签名。


### 交易的合法性

交易是否合法，主要由输入项的验证结果决定：解锁数据+锁定脚本的执行判定，以及签名的合法性。

但输出项的合法性也会影响交易的验证结果：

- 币金花费：输出项的币金数量必须大于零，且全部币金输出项的总额不得超过输入项的币金总额。
- 安全限制：输出项中各字段值的长度必须满足系统设计的限制要求（如标题长度、内容长度、脚本长度等），否则验证失败。



## 附：铸币交易（Coinbase）

铸币交易是一笔特殊的交易，它没有输入源（即没有花费项来源），是币金从无到有凭空生成的创造者。

作为基础性的设计，它需要包含如下关键信息：

- 区块高度：明确Coinbase交易的位置。
- 择优凭证：区块铸造者证明，也用于分叉之后的链段竞争验证和对比。
- 收益分成：铸凭者（铸凭交易提供者）、校验组、Blockqs、Depots、STUN服务等。
- 自由数据：类似于Bitcoin，为铸造者提供一定的灵活性（<256字节）。
- 币金销毁：交易费50%的销毁量（单位：chx）。注意应与收益总额中未销毁的交易费耦合。

> **注：**
> 铸币交易必须始终位于区块收集的交易序列的首个位置（[0]）。
> 铸币交易没有输出项配置，全部为币金输出。


### 收益分成

**收益总额 = 铸币 + 交易费（未销毁部分） + 兑奖截留**

在 Coinbase 中，收益分成表现为 5 笔输出。按顺序分别为：铸凭者、校验组、Blockqs、Depots、STUN。

5笔输出的金额按设计比例分配，总额必须等于收益总额。

```go
// 交易费销毁计算：
销毁额 = 交易费总额 * 50%（向下取整）
未销毁交易费 = 交易费总额 - 销毁额
```


#### 铸凭者&校验组

铸凭者和校验组的花费无需经过兑奖槽位的认可，直接使用即可。但通常会等到31个确认之后才使用（无分叉之忧）。

锁定脚本很简单：一个 `SYS_CHKPASS` 即可。


#### Blockqs、Depots、STUN

三个公共服务的奖励需要后续区块铸造者的认可标记（兑奖槽位），锁定脚本必须包含 `SYS_AWARD` 指令。

```go
SYS_CHKPASS                     // 系统内置验证（公钥匹配，签名通过）
SYS_AWARD[<index>, <Height>]    // 注意两个附参必须正确设置
PASS                            // 兑奖验算通过
```



## 附：公钥哈希的地址编码

将公钥哈希采用某种规则编码为可读的文本形式，即为公钥地址。

> **附注：**
> 公钥哈希较长，可能采用Base58编码方式，尽可能友好人类视觉。
> 这可能不太友好听觉，但……


### 编码

0. 公钥哈希添加识别前缀，即“前缀 + 公钥哈希”。
1. 执行2次哈希运算，取末尾4字节为 **校验码**。哈希算法：`SHA2-256`。
2. 在公钥哈希之后附上校验码（*此时无前缀*），编码为文本序列。
3. 附上识别前缀，即“前缀 + 文本序列”即为公钥地址。


### 校验

0. 提取前缀和文本序列。
1. 将文本序列解码为字节序列。截取末尾4字节校验码，前段即为公钥哈希。
2. 公钥哈希前置识别前缀（即“前缀 + 公钥哈希”），执行哈希运算取末尾4字节为校验码。
3. 比较两个校验码，相同则通过。



## 附：交易的存储与验证

交易数据的完整存储是由外部的第三方公共服务提供的，但原则如下：

- 数据部分：以交易年度（按交易时间戳计算）和交易ID为索引，存储交易的完整数据。由需要数据的用户使用。
- 验证部分：以区块年度和高度为索引，存储区块内交易ID清单，可提供交易的哈希校验路径。

> **注：**
> 用户拥有区块头链（验证总根），可以按区块高度验证交易数据。

具体设计详见：[cxio/blockqs](https://github.com/cxio/blockqs) 项目。
