## 批量付款確認

### 用例

前期支付不可花費，需要等待後期的一次確認。等待確認的支付可以不止一個，但都用同一個確認。
（註：前期支付後，付款者就不能再使用那筆資金了）

**條件：**
> 付款和收款雙方共同構造一個雙重簽名地址（1/2模式），任一簽名皆可使用。<br>
> 雙重簽名地址的公鈅哈希序列假設為「付收」序（[0] = 付款者，[1] = 收款者）。<br>


### 原理

- 前期的支付是有效付款（已進入區塊），但是腳本中存在一個哈希屏障，而哈希源有後期的確認交易提供。
- 為了可以取消前期的支付，需要植入一個過期贖迴的邏輯。


### 腳本

**前期支付交易：**

```go
// 解鎖參數
// <sig> <pubKey> <pkHash>... // 繫統內置驗證方式（略）
// 確認交易定位信息：
// 理想塊高度、交易ID、輸出索引。收款者提供。
<Height> <TxID> <OutIndex>

// 鎖定腳本
TIME{Stamp} {<expireTime>} GT       // 是否過期
IF{                                 // 是：
    MULSIG[0] PASS                  // 由付款方簽名，贖迴通過
    EXIT                            // 結束腳本
}                                   // 否：
JUMP                                // 跳轉取值（哈希源）
FN_HASH256 DATA{<hashResult>} EQUAL // 哈希屏障
MULSIG[1]                           // 是否由收款方簽名
BOTH PASS                           // 前兩者都為眞時通過（收款方使用轉移）
// @POPS[2] EVERY PASS              // 效果同上
```


**後期確認交易：**（輸出類型字段置位「中介管理」）

```go
{<hashSource>}  // 簡單提供哈希源卽可（需由 JUMP 指令跳轉）。
```


### 優點

支付實際上已經發生，付款者的幣金已經轉移（公共可見）。收款者隻是因為暫時無法使用，而讓幣金被凍結了。這樣的凍結可以給付款者一個審核所有前期支付的機會，預防出錯或滿足其它需要。這在某些場景下可能很有用。

> **註意：**<br>
> 為了讓收款者有充分的時間進行幣金轉移，確認交易應當儘量提前，或有效期足夠長。<br>
> 因為有效期過後，付款者已經可以贖迴，雖然收款者依然可以繼續使用，但這合常理嗎？<br>
