## 链间兑换

两条区块链上的币金可以相互兑换：*不是币金跨链支付，而是两对账户本链转账*。

- 假设甲有A链上的币金 `a1`，乙有B链上的币金 `b1`。
- 甲提供一个B链上的收款地址 `B2`，乙将 `b1` 发送到 `B2`。
- 相应的，乙提供一个A链上的收款地址 `A2`，甲发送 `a1` 到 `A2`。

**假设：**
- 甲有**A链**上的地址 `甲A` 和**B链**上的地址 `甲B`。甲**A链**上有钱。
- 乙有**A链**上的地址 `乙A` 和**B链**上的地址 `乙B`。乙**B链**上有钱。

现在，甲需要用A链上的钱和乙B链上的钱进行兑换：
- `甲A` => `乙A`
- `甲B` <= `乙B`


### 准备

甲乙共同构造A链和B链上的多重签名收款地址 `AA` 和 `BB`，签名模式为 `1/2`（任一签名即可），锁定脚本包含一个哈希屏障和时间限定。

假设多重签名地址的公钥哈希顺序为`「甲乙」`，即：[0] = 甲，[1] = 乙。


### 流程

1. 甲先在**A链**支付：`甲A => AA`，设置一个哈希屏障和过期赎回分支。
2. 乙看到该交易后，在**B链**支付：`乙B => BB`。设置相同的哈希屏障，同样包含过期赎回。

等待两笔交易确认……然后：

1. 甲先在**B链**转出：`BB => 甲B`，解锁需提供哈希源（`<hashSource>`），即向乙公开。
2. 乙获取哈希源（`<hashSource>`），然后在**A链**转出：`AA => 乙A`。兑换完成。


### 脚本参考

#### 甲：A链兑付脚本（`甲A => AA`）

```go
// 解锁数据
// {
//     {2}                          // 验证类型（多签）
//     <flag>                       // 授权标记
//     <[sig]>                      // 签名数据
//     <[pubKey]>                   // 公钥序列
//     <[pubHash]>                  // 公钥哈希（地址）序列
// }>> 载入环境
//
// 解锁脚本
// 乙转移A链币金时提供。
DATA{<hashSource>}                  // 哈希源，甲方创建并保留

// 锁定脚本
// 此为交易中的输出脚本，公开可见。
SYS_CHKPASS                         // 内置验证（从环境读取实参）

ENV{Height} {<expireHeight>} GT     // 是否过期
IF{                                 // 是：
    SIGNED[0]                       // 是否由甲签名，返回Bool
    PASS                            // 栈顶为真时通过（甲赎回）
    EXIT                            // 结束脚本
}                                   // 否：
SIGNED[1] PASS                      // 是否由乙签名，真时通过
FN_HASH256                          // 栈顶剩余项为<hashSource>，计算哈希
DATA{<hashResult>}                  // 预置哈希结果（哈希屏障）
EQUAL PASS                          // 栈顶两个序列相等比较，真时通过（乙转出）
```


#### 兑换模型

甲创建的兑换脚本需要符合预定的格式，才能保证乙正常地转出币金。

这种预定的格式由兑换模型来保证。

乙检查甲的兑换脚本是否符合格式，并提取时间（高度）和哈希屏障数据，构建B链上的兑换脚本。

```go
MODEL{                              // 模式匹配
    SYS_CHKPASS
    ENV{Height} ^{?} #[32] GT       //1: 提取高度（时间）
    IF{                             // if指令，进阶匹配
        SIGNED[0] PASS              // 严格匹配
        EXIT                        // 严格匹配
    }
    SIGNED[1] PASS                  // 严格匹配
    FN_HASH256                      // 严格匹配
    ^DATA[32]{?} #[32]              //2: 提取32字节的哈希结果（哈希屏障）
    EQUAL PASS                      // 严格匹配
}                                   // => ([expireHeight, hashResult], Bool)
PASS                                // 匹配是否通过
@POP OUTPUT                         // 导出提取的两个值
BUFDUMP                             // 触发外部监听（乙构建B链的兑换脚本）
```


#### 乙：B链兑换脚本（`乙B => BB`）

```go
// 解锁数据
// {
//      {2}                         // 验证类型（多签）
//      <flag>                      // 授权标记
//      <[sig]>                     // 签名数据
//      <[pubKey]>                  // 公钥序列
//      <[pubHash]>                 // 公钥哈希（地址）序列
// }>> 载入环境
//
// 解锁脚本
// 在甲转移B链币金时提供。
DATA[32]{<hashSource>}              // 哈希源，甲转出时公开

// 锁定脚本
SYS_CHKPASS                         // 内置验证（从环境读取实参）

ENV{Height} {<expireHeight2>} GT    // 是否过期
IF{                                 // 是：
    SIGNED[1] PASS                  // 由乙签名，通过（赎回）
    EXIT                            // 结束脚本
}                                   // 否：
SIGNED[0] PASS                      // 是否由甲签名，真时通过
TOP                                 // 栈顶为<hashSource>，引用（复制）
FN_HASH256                          // 计算<hashSource>的哈希
DATA{<hashResult>}                  // 哈希结果预置（屏障）
EQUAL PASS                          // 栈顶两个序列相等比较，真时通过（甲转出）

@POP OUTPUT                         // 导出栈顶<hashSource>
BUFDUMP                             // 触发外部监听（乙：及时转出AA地址币金）
```

> **提示：**
> 甲应当先在本机私有环境下尝试执行脚本，如果成功才对外发布。


### 注意

这只是一个概略的兑换脚本，如果两条链拥堵，交易迟迟不能确认，会带来问题。

一个可能的解决方式是创建更复杂的约束，或者简单允许赎回期后，对方依然可以转出（用 `SWITCH{}`），不过如此就有竞争关系了。
