## 選舉子鏈腳本集

創始區塊裏包含了起始交易，起始交易裏包含了授權簽發投票權和驗證票的公鈅清單。

客戶端程序裏有一個固定規則：隻有創始區塊裏起始交易列出的公鈅清單有權創建憑信資產（投票權、驗證票），繫統禁止空白輸入的憑信資產創建（卽隻能轉移/使用）。同時客戶端會維護一個 UTXO 集，裏面包含了子鏈自身發行的激勵幣金和憑信資產（投票權、驗證票）。

投票權和驗證票資產可以被花費（使用），使用次數被設置為 `1`。

> __註：__
> 起始交易在主鏈上隻是一筆 __證據__ 交易。


### 投票權

投票的憑信資產可以被轉移，但轉移次數為 `1`，使用投票權的交易應當是內容票（輸入項）。

除了基礎性的用戶簽名驗證外，投票權的輸出腳本中包含了對內容票提交內容的格式驗證。這是必要的，它旣限定了用戶的類型，也避免了用戶提交無效的內容。也因此，選票內容的條目和格式需要在啓動子鏈網絡前確定。

另外投票權憑證的標題也可以幫助用戶辨別自己是投票權而非驗證票。

> __註：__<br>
> 內容票中隻對條目的內容加密而非整體加密，條目名稱留待格式驗證用。<br>
> 如果選票內容格式需要後期才能確定，則輸出腳本的驗證部分可以引入一個內容票約定標識（避免被確認票使用）。<br>

```go
// 解鎖腳本
// <sig> <pubKey> <pkHash>...       // 繫統內置驗證方式，略
TEXT[...]{<conNames>}               // 投票內容條目名稱序列。註：...指具體的長度值，<itemNames>為指代
DATA[...]{<conData>}                // 投票內容值數據（已加密）

// 鎖定腳本
// 僅附加格式驗證（前有繫統內置驗證）
@MODEL<>{                           // 內容格式檢測，非法則失敗
    TEXT[...]{<itemNames>}          // 條目名稱序列嚴格匹配
    DATA[?]{?}                      // 匹配任意字節序列，長度不定（卽加密數據）
    #[4]                            // 提取前面匹配的加密數據（開票協助）
}
OUTPUT BUFDUMP                      // 導出前面提取的數據並轉儲。僅開票階段需要，空執行無害
```

> __註：__<br>
> 代碼中的換行隻是一種視覺友好，以便邏輯清晰（同行上的指令邏輯更緊密些）。<br>
> 轉儲是一個外部激發機製，節點運行腳本時需先註冊處理器，數據會傳遞給處理器來完成外部的邏輯。<br>


### 驗證票

驗票憑信也可以被轉移，同樣的轉移次數為 `1`，使用驗證票的交易為內容票之後的確認票（輸入項）。

除了基礎性的用戶簽名驗證外，驗證票的輸出腳本中包含了由登記號哈希和口令哈希的哈希屏障驗證，以及內容票引用和登記號哈希數據項的導出。導出和轉儲是無阻塞執行的，如果運行腳本的節點先期註冊了處理器，這些數據就會被傳遞過去（由另外的監聽服務提供銜接）。這樣就可以執行開票環節了。

```go
// 解鎖腳本
// <sig> <pubKey> <pkHash>...       // 繫統內置驗證方式，略
{<conBlockNum>}[4]                  // 內容票所在區塊號（4字節整數）
DATA[32]{<conTxID>}                 // 內容票交易ID
DATA[32]{<regNumHash>}              // 登記號哈希
DATA[32]{<passwdHash>}              // 用戶口令的哈希

// 鎖定腳本
@TOPS[2]                            // 引用棧頂2項（<regNumHash>和<passwdHash>）（註：不彈出）
ADD                                 // 連接實參空間兩項數據（登記號哈希+口令哈希）
FN_HASH256                          // 計算哈希結果
DATA{<hashResult>} EQUAL            // 壓入腳本中已有的結果序列，比較兩個序列是否相同
PASS                                // 前面的比較相同則通過

POP                                 // 彈出棧頂1項（卽口令哈希），丟棄
@POPS[3]                            // 彈出棧頂3項（區塊號、交易ID、登記號哈希）到實參空間
OUTPUT BUFDUMP                      // 導出實參空間的3項值並轉儲（開票階段需要）
```

> __註：__
> 腳本中值指令的值和指令計算的值都會自動壓入數據棧，註釋中並不特別說明。


### 內容票

內容票會花掉投票權交易的憑信，使用次數遞減為 `0`，因此內容票的憑信將不能再被轉移使用。輸出腳本無意義（可能設置為空）。

使用投票權時，內容票需要提交預定條目格式的內容，條目的值需要被加密。註意輸入項是投票權交易的輸出，通常來說，如果引用錯誤（使用了驗證票），輸入腳本的驗證是沒法通過的（也卽無效），這不會帶來問題。

```go
// 輸出腳本為空
NIL
```


### 確認票

確認票會花掉驗證票交易的憑信，使用次數遞減為 `0`，同內容票一樣，確認票也是一次性使用的。輸出腳本無意義。

使用驗證票時，確認票需要提供領票時獲得的登記號哈希、口令哈希以及內容票引用（區塊號和交易ID）。前兩者用於驗證哈希屏障，內容票信息用於檢索讀取內容，另外登記號哈希還用於構造解密密碼（`密碼 = Hash(基碼 + 登記號哈希)`）。

基碼是由客戶端私有導入計算環境的，驗證腳本執行時會導出登記號哈希和內容票引用信息。這樣客戶端就可以匯集信息構造密碼，檢索內容票解密讀取了。

```go
// 輸出腳本為空
NIL
```


### 附註

#### 開票

節點運行的客戶端App先註冊開票處理器，導入基碼，再次執行確認票驗證，提取數據並運行開票處理器。

計票原理&過程：

- 確認票的輸入驗證腳本（卽驗證票輸出腳本）會導出3項數據：內容票的 __區塊號__、__交易ID__ 和選民的 __登記號哈希__。
- 根據內容票引用信息，開票處理器請求客戶端App執行內容票驗證，獲得內容票導出的 __加密數據__。
- 開票處理器構造解密密碼（`Hash(基碼+登記號哈希)`），解密內容票數據，讀取投票內容、統計、發布。


#### 安全性

選舉鏈上的憑信轉移實際上隻有一次，兩個層級：

1. 機構簽發的選舉權證（投票權、驗證票）。根據起始交易裏的授權公鈅受限創生。
2. 選民投票用的使用權證（內容票、確認票）。花費掉上面機構簽發的選舉權證，自身不能再被花費。

所以客戶端在開票時還可以檢査 __驗證票__ 的輸入項是否源於起始交易。雖然在繫統禁止空白來源憑信創建的情況下，這種檢査並不必需。
