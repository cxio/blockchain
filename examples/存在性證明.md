## 存在性證明

### 用例

對於一筆交易來說，理論上時間戳是任意的，它可以由用戶自由設置。這可能帶來一個缺陷：你無法嚴格區分兩筆交易在現實中誰更先出現。這在存在性證明中是一個嚴重的問題。例如「原創論文」證明：

- 一個竊取者可以對一個出現不久的論文證明交易復製其證明內容，脩改版權，創建自己的交易並設定為更早的時間戳。
- 通常原創者的交易會先進入區塊，因為它確實先出現，但這不足以證明竊取者的交易是後出現的，因為網絡可能擁堵延遲。

未收錄交易的「交易過期」特性設計，使得這一問題可以得到輔助解決。

> **附：交易過期**
>
> 未被收錄的交易超過一定時間後會作廢，這個期限可能是3天（720區塊）。過期是按交易的時間戳和當前時間對比判斷的。它的意義在於：<br>
> 1. 縮減未確認交易的規模。<br>
> 2. 提昇時間因子的價值，為某些應用提供條件。<br>
>
> 人們不應當期待一筆超過3天都未完成（確認）的交易依然有效，反而，它可能造成一種漫長期待的負面效應。<br>


### 操作

分兩次發布論文的存在性證明。

1. 第一次發布一筆結果聲明交易（管理者），提供論文聲明（和可能的附件ID）的哈希摘要。
2. 交易進入區塊並等到過期時段後，發布正式的論文聲明（證據輸出），並跳轉到前一筆交易驗證，獲得證明。


### 腳本

**結果聲明：**（管理者交易）

```go
IN{Content}                     // 取證據內容
IN{Attachment}                  // 取證據附件ID
ADD                             // 串接
FN_HASH256                      // 計算哈希摘要
DATA{<hashResult>} EQUAL PASS   // 是否與結果相等而通過驗證
```

**論文聲明：**（證據交易）

```go
{<Height>}              // 理想塊高度
DATA{<TxID>}            // 交易ID
{<OutIndex>}            // 輸出序位（下標）
JUMP                    // 跳轉到結果聲明驗證

@TEXT{Is good.} PRINT   // 友好打印

@ENV{TxID}              // 取交易ID
@IN{Title}              // 取證據標題
OUTPUT BUFDUMP          // 導出必要信息，通知外部應用
```

> **說明：**<br>
> 在證據輸出中寫入論文的標題、內容和附件ID，由跳轉目標提取、計算和驗證。<br>
> 因為證據信元沒有轉移邏輯，所以其腳本需要由應用App主動處理。<br>
