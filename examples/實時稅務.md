## 實時稅務

**註**：假設為一種新的監管方式。


### 前言

一個實體向企業支付的幣金腳本中，可以包含財務匯報的邏輯。這些幣金不是在企業收款時主動匯報，而是在使用時由代碼中的匯報邏輯強製執行：當輸出作為新交易的輸入時腳本代碼被執行。

財務匯報的邏輯並不由企業自己掌控，它們由支付幣金給企業的實體書寫，因此除非合謀，理論上企業無法作弊。


### 企業身份

從數字安全的角度來說，企業的身份需要認證，它可能由企業的官方簽名來實施。因此財務匯報還需要企業提供簽名來確認。

給企業支付幣金的實體無法知道企業情況，當然也無需置入企業簽名的驗證代碼，但是需要提供簽名數據的跳轉邏輯：跳轉到財務匯報管理機構預先發布的管理者交易。這裏，接收財務匯報的管理機構是已知的，那些管理者交易也是已知的。


### 腳本

#### 檢査財務信息的監管腳本

攜帶監管腳本的管理者交易需要先發布，用於適用企業的財務匯報。這樣的監管腳本通常有足夠長的有效期。

假設該管理者交易被收錄在第 `57043` 號區塊，交易ID為 `0xa1075db55d416d3ca199f55b6084e2115b9345e16c5cf302fc80e9d5fbf5d48d`，監管腳本為交易中唯一的輸出項（卽腳本在輸出集中下標為 `0`）。

```go
@POPS[2]            // 提取傳入的2項參數
OUTPUT BUFDUMP      // 導出並觸發監管服務器上的App，執行驗證和登記等相關工作

TRUE PASS           // 允許企業花費（無條件通過）
```

> **附註：**<br>
> 監管腳本通常是獨立的邏輯，由GOTO指令跳轉過來，需要處理PASS邏輯。<br>


#### 嵌入監管邏輯的付款腳本

```go
// 解鎖參數
// <sig> <pubKey>...            // 繫統內置驗證方式，略
// <sigCorpData>                // 企業簽名數據，企業自己提供

// 鎖定腳本
IN{Amount}                      // 花費的幣金數量
@POPS[2]                        // 提取企業簽名數據和轉出金額
GOTO[57043, 0xa1075db5..., 0]   // 跳轉到上面的管理者交易的監管腳本

// 花費許可將由監管腳本控製，
// 因為監管腳本已知，所以許可情況明確。
```


### 備註

**這並不是企業傳統報稅的流程，而僅是假想了一種可能，說明某種邏輯上的可行性。**
