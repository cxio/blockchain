## 網購支付流程

### 用例

> **購買**：客戶購買商傢的商品，付款但無法被取走，取貨後完成支付。<br>
> **退貨**：商傢支付返還金額，客戶提取返迴金額，商傢提取購買金額。<br>


### 購買

#### 流程

1. **購買**：客戶付款給商傢的支付輸出包含一個哈希屏障，商傢沒法立卽取款使用。付款暫時凍結。
2. **物流**：商傢發貨，給物流支付的運費裏包含同樣的哈希屏障。運費暫時凍結。
3. **簽收**：物流送達客戶，客戶驗貨簽收，提供哈希源。
4. **結賬**：物流提取運費的交易廣播，商傢獲取哈希源，提取購買支付的金額。

#### 腳本

**購買支付：**

```go
// 解鎖參數
// <sig> <pubKey>...                // 繫統內置驗證方式，略
<hashSource>                        // 哈希源，客戶創建並保留

// 鎖定腳本
FN_HASH256 DATA{<hashResult>} EQUAL // 哈希屏障
BOTH PASS                           // 前兩者都為眞時通過
```

**運費支付：**

```go
// 解鎖參數
// <sig> <pubKey>...                // 繫統內置驗證方式，略
<hashSource>                        // 哈希源，客戶提供

// 鎖定腳本
$TOP                                // 信息保留
FN_HASH256 DATA{<hashResult>} EQUAL // 哈希屏障
BOTH PASS                           // 前兩者都為眞時通過

$[0] OUTPUT BUFDUMP                 // 導出哈希源，商傢監聽提取
```


### 退貨

#### 流程

1. **退款**：商傢與客戶協商返還金額，商傢創建返還交易，包含同樣的哈希屏障。
2. **取款**：客戶提取返還金額，公開哈希源。
3. **結賬**：商傢獲取哈希源，提取購買金額。


#### 腳本

**退款支付1：**

```go
// 與運費支付腳本相同，隻是收款者為客戶，略。
```

該模式有一個潛在問題：如果客戶中途變卦不收退款，則商傢也無法迴收資金。這是一個雙輸的局面，但從某種角度看，這對商傢更不利。商傢的幣金保有量會受製於良莠不齊的客戶的直接影響，商傢迴收資金的壓力明顯更大。

**退款支付2：**

為了防止這一情況發生，退款支付可采用延時贖迴模型。協商一致之後的退款，客戶需要及時取出。

```go
// 解鎖參數
// <sig> <pubKey> <pkHash>...       // 繫統內置驗證方式，略
<hashSource>                        // 哈希源，客戶提供

// 鎖定腳本
$TOP                                // 信息保留
TIME{Stamp} {<expireTime>} GT       // 是否過期
IF{                                 // 是：
    MULSIG[0] PASS                  // 由商傢簽名，贖迴通過
    EXIT                            // 結束腳本
}                                   // 否：
MULSIG[1]                           // 由客戶簽名
FN_HASH256 DATA{<hashResult>} EQUAL // 哈希屏障
BOTH PASS                           // 前兩者都為眞時通過

$[0] OUTPUT BUFDUMP                 // 導出哈希源，商傢監聽提取
```

> **註：**<br>
> 商傢和客戶需要共同構建一個 `1/2` 模式的雙簽名收款地址。<br>
