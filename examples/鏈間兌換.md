## 鏈間兌換

一條區塊鏈上的幣如果能自動兌換為另一條鏈上的幣，有時會非常有用。

區塊鏈上的幣流通是鏈局限的（僅限於本鏈環境），因此鏈間兌換並不是把一條鏈上的幣變成另一條鏈上的幣，隻是**兌換**而已。卽假設你有比特幣，希望把一些比特幣換成以太幣，此時需要有一個人願意接收你的比特幣，同時他也有以太幣。如果這種交換可以自動實現（無需第三方交易所），就是鏈間兌換。

通過鏈間兌換協議，你可以很安全地把你的比特幣發送給他，而他也可以很安全地把以太幣發送給你，雙方都無法欺騙。

> **註：**
> 上面舉例的比特幣和以太幣隻是便於理解，目前它們並不具有鏈間自動兌換的能力。


### 用例

甲有A鏈上的幣金 `a1`，乙有B鏈上的幣金 `b1`。甲向乙請求兌換B鏈上的幣金。

**條件：**
> 甲乙共同構造A鏈和B鏈上的多重簽名地址 `a2` 和 `b2`，簽名模式為1/2（任一簽名卽可使用）。<br>
> 構造多重簽名地址的公鈅哈希序列假設為甲乙序（卽：[0] = 甲，[1] = 乙）。<br>


### 過程

1. 甲先發送A鏈上的幣金支付交易 `txA: a1 => a2`，設置一個哈希屏障和過期贖迴分支（`IF{...}`）。
2. 乙看到交易 `txA` 後，發送B鏈上的幣金支付交易 `txB: b1 => b2`，前置相同的哈希屏障，同時也包含過期贖迴邏輯。
3. 等到兩筆交易確認後，甲轉移 `txB` 中的幣金。因為哈希屏障的存在，甲必須提供哈希源（公開）。
4. 甲的轉移交易會觸發哈希源導出，乙可事先註冊中間件，自動轉移 `txA` 中的幣金。兌換完成。

**註意：**
> 雙方應當在贖迴期限之前（有效期內）完成幣金的轉移。<br>
> `txB` 的有效期應當比 `txA` 的短，如 `txA` 為2小時，`txB` 可為1小時。這樣當甲轉移 `b2` 幣金後，乙有足夠的時間轉移 `a2` 幣金。<br>
> 較長的有效期是一種安全性保障，如果彼此都是及時轉移，兌換就會很快完成。<br>


### 腳本

#### 甲發起的兌換請求

**A鏈：`txA: a1 => a2`**

```go
// 解鎖腳本
// <sig> <pubKey>...                // 繫統內置驗證方式，略
DATA[32]{<hashSource>}              // 哈希源，甲創建並保留

// 鎖定腳本
SYS_TIME{Stamp} {<expireTime>} GT   // 是否過期
IF{                                 // 是：
    MULSIG[0] PASS                  // 由甲簽名，通過（贖迴）
    EXIT                            // 結束腳本
}                                   // 否：
MULSIG[1]                           // 是否由乙簽名
@PEEK[-2] FN_HASH256                // 引用棧頂第2項，計算哈希
DATA{<hashResult>} EQUAL            // 哈希屏障，比較結果
BOTH PASS                           // 前兩者都為眞時通過（乙轉移）
```

> **提示：**<br>
> 腳本代碼中的指令用空格和換行符分隔，兩者沒有區別。邏輯上更緊密的會放在同一行而便於理解。


**兌換模型：**

甲創建的兌換支付腳本需要符合預定的格式，才能保證乙能夠正常地提取幣金轉移。這種預定的格式由兌換模型來保證。
乙檢査甲的兌換腳本是否符合格式，提取時間戳和哈希屏障數據，構建兌換支付交易。

> **註：**<br>
> 兌換模型是一段滿足兌換邏輯的預定義腳本，通常會被共享發布出來。腳本是可執行的，待檢測腳本是它的實參。<br>

```go
@MODEL<>{
    SYS_TIME{Stamp} {?} #[4] GT     //1: 提取時間戳，{?} 同 ? {}
    IF{                             // if指令，進階匹配
        MULSIG[0] PASS              // 嚴格匹配
        EXIT                        // 嚴格匹配
    }
    MULSIG[1]                       // 嚴格匹配
    @PEEK[-2] FN_HASH256            // 嚴格匹配
    DATA[32]{?} #[4] EQUAL          //2: 提取32字節的哈希結果
    BOTH PASS                       // 嚴格匹配
}
OUTPUT BUFDUMP                      // 導出MODEL提取的值，觸發外部監聽（乙構建 txB 交易）
```


#### 乙創建的兌換支付

**B鏈：`txB: b1 => b2`**

```go
// 解鎖腳本
// <sig> <pubKey>                       // 繫統內置驗證
DATA[32]{<hashSource>}                  // 哈希源，甲轉移時提供

// 鎖定腳本
&TOP                                    // 引用棧頂哈希源存儲到局部域
SYS_TIME{Stamp} {<expireTime2>} GT      // 是否過期
IF{                                     // 是：
    MULSIG[1] PASS                      // 由乙簽名，通過（贖迴）
    EXIT                                // 結束腳本
}                                       // 否：
MULSIG[0]                               // 是否由甲簽名
&[0] FN_HASH256                         // 引用哈希源，計算哈希值
DATA{<hashResult>} EQUAL                // 哈希屏障，比較結果
BOTH PASS                               // 前兩者都為眞時通過（甲轉移）

&[0] OUTPUT BUFDUMP                     // 導出哈希源，觸發外部監聽（乙對 txA 的轉移）
```

> **註意：**<br>
> 甲應當先在本機私有的環境下提供解鎖參數執行腳本，如果可以成功轉移幣金，則發布為正式的交易。<br>
> 甲在B鏈上的幣金轉移也可能是自動的，但監聽 `b2` 收款地址的情況是一種外部行為。<br>


### 安全性

交易擁堵可能導致兌換的轉移交易不能及時確認，如果超出有效期，贖迴轉移交易按理來說就可以正常通過驗證了。這樣，兌換轉移和贖迴轉移都有效，雙花的局面就出現了。

但這裏其實潛藏着一個安全保障，兌換轉移的交易時間戳必然先於贖迴轉移，因此實際上贖迴轉移交易沒法廣播出去，這由零確認的「時序保障」規則實現。如果兩個有效期的時間差距足夠，就可以最大限度地避免交易擁堵而造成這樣的狀況。

**風險：**
贖迴轉移交易可能會避開時序保障規則，但這需要攻擊者自己控製足夠的節點來違反該規則，同時支付高昂的交易費來賄賂鑄造者（或自己就是鑄造者）。這種風險是在上面的雙花情況發生時才存在。


### 備註

這裏並不涉及幣金兌換的比率問題，隻要雙方預先約定卽可。這裏的兌換邏輯存在相互製約，雙方無需事先信任。
