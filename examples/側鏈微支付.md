## 側鏈微支付

### 前言

一條區塊鏈通常會自己創生本鏈的幣金（無中生有），但這並非絕對，以另一條鏈上的幣金為依據，通過某個匯率轉換而來也是可行的。如果我們把轉換來源的鏈稱為主鏈，則轉換生成幣金的鏈可稱為側鏈。主鏈上通過充值提供側鏈幣金生產的依據，這就是側鏈微支付的原理（有點類似第三方支付）。

與傳統的法幣用於購買商品不同，側鏈微支付的粒度是任意的，它可以表達非常寬泛的支付邏輯。實際上，隻要涉及價值或信用的傳遞，粒度任意的微支付就可以有用武之地，比如將閱讀量進行價值量化，一個超大的匯率如 `1:1,000,000` 也是可能的。


### 微支付的側鏈

微支付繫統通常需要足夠快的確認時間，一個擁有大量用戶的微支付繫統會產生規模龐大的數據，因此它可能很難成為一個廣域範圍的通用繫統。這裏假定微支付的側鏈用於一個局部的微網，比如一個商場、連鎖酒店、愛好者俱樂部或者是一個App內容平臺。

微支付的側鏈可能並不需要是廣域的，因為廣域的邏輯已經由主鏈來擔當了。主鏈與側鏈之間是一個「充值-鑄幣」和「提現-銷毀」的邏輯。也正因如此，作為優化，一種定期清轉歷史數據，讓側鏈輕鬆上綫的「重啓」模式是可能的。

#### 區塊鑄造與鑄幣的分離

微支付的側鏈由主鏈上的一筆創始交易開啓，交易裏定義了鑄幣者身份，由一個不可脩改的憑信表達（可能包含匯率配置）。側鏈的創始區塊裏包含了這筆主鏈上的創始交易，側鏈上也有憑信的邏輯，鑄幣憑信轉移可以讓鑄幣者產生變化，所以安全性可以保證。不可變憑信采用引用傳遞，對鑄幣者的驗證可以定位到創始區塊裏的創始交易，因此非法的鑄幣可以很容易辨別。

與普通的區塊鏈不同，側鏈裏的鑄幣交易（CoinMake）由鑄幣者創建而不是由區塊的鑄造者創建，因此這裏的區塊鑄造和鑄幣需要分離。當然，區塊鑄造者（Coinbase）依然擁有收取交易費的權利，而鑄幣交易就隻是一筆特別的交易而已。

邏輯上，微支付的側鏈不是去中心化的，因為它存在一個兌換主體，同時也需要擔負責任。但側鏈的運行可以是去中心化的，激勵自由的參與者，可以大幅地節省成本並提高安全性。


### 用例

> **註：**<br>
> 這裏不涉及側鏈本身的實現細節，隻是假定它與主鏈有相似的能力。<br>

假設存在一個微支付實體MPC，他們用一種代幣 `RC` 來鼓勵用戶對閱讀作筆記。當用戶發布的筆記被其他用戶點讚後，MPC會對讚的情況進行評估和打賞，打賞的介質就是 `RC`。

因為 `RC` 是一種區塊鏈代幣，流通是它的基本能力，因此擁有 `RC` 的用戶也可以參與到遊戲中來，或者打賞，或者用 `RC` 在繫統內購買小商品小禮品等。於是，這個閱讀繫統裏便創建出了一個小生態。

為了簡化 `RC` 與外部世界的價值溝通，MPC決定采用某流行主鏈的幣金 `PC` 為自己的代幣轉換源。任何一位用戶通過充值 `PC` 兌換成 `RC`，卽可進入到這個「互動閱讀」微繫統裏來，而如果想要離開，簡單地將 `RC` 提現為主鏈幣金 `PC` 卽可。

這裏假設 `PC` 與 `RC` 的兌換比率是固定的 `1:10000`。采用標準的「鏈間兌換」模式實施。

#### 兌換主體

卽經營該微繫統的實體MPC，他們在主鏈上擁有接收充值的地址或地址集，在側鏈上則為鑄幣者，以及接收 `RC` 提現並返還主鏈幣金 `PC` 的管理者。這裏假設MPC有一個收款地址 `Maddr`，用於收集各充值地址的幣金轉移。


### 充值兌換

用戶充值的目標地址並不是 `Maddr`，每個用戶的充值地址都可能不一樣，但充值會被轉移到目標收款地址。充值用戶實際上也可以限定收款地址（通過 `OUT` 指令），但這隻能是一種象徵性的說明（因為很難同時限定金額）。


#### 主鏈充值

示意：{UserAddrA} => tmpAddr >> {Maddr}

```go
// 解鎖參數
// <sig> <pubKey> <pkHash>...
<outIndex>                          // 收款地址所在輸出項下標，MPC提供
<hashSource>                        // 哈希源，由用戶創建

// 鎖定腳本
TIME{Stamp} {<expireTime>} GT       // 是否過期
IF{                                 // 是：
    MULSIG[0] PASS                  // 由用戶簽名，贖迴通過
    EXIT                            // 結束腳本
}                                   // 否：
MULSIG[1]                           // 是否由MPC簽名
FN_HASH256 DATA{<hashResult>} EQUAL // 哈希屏障
OUT{Receiver} DATA{<Maddr>} EQUAL   // 限定收款地址

@POPS[3] EVERY PASS                 // 前3項判斷都為眞時通過（MPC轉移）
```

**說明：**
- 哈希結果（hashResult）通過模式匹配提取，側鏈兌換中轉地址和充值金額通過腳本外方式獲取。
- 限定收款地址可以表達一種公開性，但腳本無法提供強製約束力。

**充值模型：**

充值腳本的格式應當符合固定模型，這樣中間件才能確定充值最終能夠轉移出去。這是一個通用結構，可以發布為一筆管理者交易共享。

```go
MODEL<PRINT>{
    TIME{Stamp} {?}[8] #4 GT            //1: 提取8字節整數時間戳
    IF{                                 // if指令，進階匹配
        MULSIG[0] PASS                  // 嚴格匹配
        EXIT                            // 嚴格匹配
    }
    MULSIG[1]                           // 嚴格匹配
    FN_HASH256 DATA[32]{?} #4 EQUAL     //2: 提取32字節的哈希結果
    OUT{Receiver} DATA[?]{?} #4 EQUAL   //3: 提取收款地址
    @POPS[3] EVERY PASS                 // 嚴格匹配
}
@POPS[3] OUTPUT BUFDUMP                 // 導出MODEL提取的值，觸發外部邏輯
```

或
```go
@MODEL<PRINT>{ /* 同上內容 */ } OUTPUT BUFDUMP
```

**說明：**
1. 提取出時間戳設定可以檢査時間長度是否足夠。
2. 提取出哈希結果用於構造側鏈兌換交易的相同設置。
3. 提取收款地址用於判斷地址是否屬於自己公布的地址集成員。


#### 側鏈兌換

示意：{鑄幣者} => tmpAddrB >> {UserAddrB}

```go
// 解鎖參數
// <sig> <pubKey> <pkHash>...
<hashSource>                            // 哈希源，用戶轉移時提供

// 鎖定腳本
$TOP                                    // 棧頂哈希源添加到變量域
TIME{Stamp} {<expireTime2>} GT          // 是否過期
IF{                                     // 是：
    MULSIG[1] PASS                      // 由MPC簽名，贖迴到一個銷毀地址（UTXO友好）
    EXIT                                // 結束腳本
}                                       // 否：
MULSIG[0]                               // 是否由用戶簽名
FN_HASH256 DATA{<hashResult>} EQUAL     // 哈希屏障（hashResult同上）
BOTH PASS                               // 前兩者都為眞時通過（用戶兌換轉移）

${TOP} OUTPUT BUFDUMP                   // 導出哈希源，便於MPC轉移用戶的充值
```

> **說明：**
> 側鏈用戶的兌換腳本無需兌換模型，用戶事前預執行成功卽可發布交易。


### 提現兌換

用戶在側鏈中向提現管理者發送側鏈幣金，同時傳遞主鏈提現地址 `withdrawAddr`。因為目標明確，提現交易通常無需贖迴，這是一個鏈間授信的邏輯。管理者接收到提現的幣金後，原則上應當銷毀它們，轉移到銷毀地址的交易會觸發主鏈的迴兌流程。


#### 側鏈提現

示意：{UserAddrB} => adminAddrB

**提現腳本1：**

簡單的提現發送。管理者銷毀該提現幣金時觸發主鏈的返還兌現，幣金數量從腳本外獲取。

```go
// 解鎖參數
// <sig> <pubKey>...    // 繫統內置驗證，略

// 鎖定腳本
DATA{<withdrawAddr>}    // 主鏈的提現地址
@POP OUTPUT BUFDUMP     // 導出主鏈地址並觸發外部邏輯
```

**提現腳本2：**

提現幣金從腳本內獲取，這需要腳本匹配一個合格的提現模型，驗證幣金數量的眞實性。

```go
// 解鎖參數
// <sig> <pubKey>...    // 繫統內置驗證，略

// 鎖定腳本
IN{Amount}              // 提現的側鏈幣金數量
DATA{<withdrawAddr>}    // 主鏈的提現地址

@POPS[2] OUTPUT BUFDUMP // 導出數據並觸發外部邏輯
```

**提現模型：**
```go
MODEL<>{                    // 無忽略指令
    IN{Amount}              // 嚴格匹配，保證數額眞實
    DATA[?]{?}              // 主鏈的提現地址
    @POPS[2] OUTPUT BUFDUMP // 嚴格匹配
}
// 這裏無需采集數據
// 當進行銷毀轉移時，腳本作為輸入被執行。
```


#### 主鏈兌迴

無特別要求，MPC用一筆普通的轉賬交易發送按匯率計算的主鏈幣金卽可。


### 固定匯率下的鑄幣平衡

理論上，如果按照固定的匯率，用戶充值後得到了數額準確的側鏈幣金，提現時反之亦然。但鑄幣交易通常也需要支付交易費，而提現後的銷毀交易也同樣如此。因此，這個微繫統實際上會慢慢變相地增發幣金。

要解決這個問題，要麽對繫統進行微調脩改，設計鑄幣和銷毀交易無需支付交易費，要麽把增發的交易費也同時銷毀。通常，後者的解決方式可能更好。


### 浮動匯率下的借貸邏輯

如果不保持固定的兌換匯率，側鏈代幣可能成為一種股票或私人債券：兌換人實際上是出借主鏈幣金給經營者，並持有其側鏈上的憑證，當約定的期限到達後，憑證提現為主鏈的幣金，完成借貸（和獲利）的返還。因為代幣可以在側鏈上流通，這能夠帶來更多的靈活性。

從長遠角度來看，借貸雙方實際上都期望主鏈幣價值穩定，如果貸方借出主鏈幣後主鏈幣昇值，實際上貸方是不樂意迴兌的，這樣可以消減主鏈幣的買盤。如果這樣的借貸關繫大量存在，就會對主鏈幣的價值穩定有正向迴饋。

借貸經營的實體通常是中心化的，身份公開已知，會受到現實世界裏的監管或約束。
