## 鏈間授信

### 用例

- 甲用戶擁有A鏈和B鏈上的地址 `甲a` 和 `甲b`，乙用戶擁有A鏈和B鏈上的地址 `乙a` 和 `乙b`。
- 甲用 `甲a` 給 `甲b` 地址授信，向乙說明 `甲b` 地址可信。
- 因為 `乙a` 可以驗證 `甲a` 的信用，所以 `乙b` 就可以相信 `甲b` 。
- 這樣 `A鏈` 就向 `B鏈` 傳遞了信用。

**說明：**
> 鏈間授信沒有雙向製約設計，這種模式僅僅是在本鏈中（可信）表達外鏈某個地址可信。<br>
> 如果需要雙向製約的交換模式，可以采用鏈間兌換流程。<br>


### 過程

1. `甲a` 向 `乙a` 發送一個數字資產 `AA`，輸出腳本中提供 `甲b` 的地址和可能需要的說明消息 `Msg`。
2. 乙偵測 `乙a` 地址發現收到了資產 `AA`，運行輸出腳本提取 `甲b` 和 `Msg`。觸發 `乙b` 向 `甲b` 地址發送資產或做其它事情。
3. 這樣，`A鏈` 上的交易行為就傳遞到了 `B鏈` 上。

> **註：**<br>
> 如果需要隱祕傳遞，甲向乙發送AA資產的交易裏 `甲b` 和 `Msg` 可以加密，加密方式需要雙方約定。<br>
> 因為AA資產交易裏沒有贖迴邏輯，所以乙並不需要構建轉移交易，但需主動運行輸出腳本以獲取相關信息。<br>
> 實際上，`甲b` 地址不必是甲自己的，它隻是甲授信的一個目標地址而已。<br>


### 腳本

**A鏈：資產轉移 `AA: 甲a => 乙a`**

```go
// 解鎖參數
// <sig> <pubKey>...    // 繫統內置驗證方式，略

// 鎖定腳本
DATA{<chainB:甲b>}      // B鏈上的甲b地址可信
DATA{<Msg>}             // 附加說明（約定格式）
@POPS OUTPUT            // 導出全部數據（僅2條）
BUFDUMP                 // 轉儲觸發外部 App(甲b, Msg)。
```
> **說明：**<br>
> 乙花費該輸出時也會有 `BUFDUMP` 觸發邏輯，但監聽是私有的，BUFDUMP非阻塞執行。<br>
> 這不會對其它公共驗證節點帶來任何問題。任何一個節點也可以監聽該事務。


**B鏈：資產轉移 `乙b => 甲b`**（或其它事情）

> **註：**<br>
> 乙註冊的監聽由另外的服務提供，應當包含了對接收地址和資產類別的判斷。<br>
> 乙運行的外部App是私有邏輯，隻是代替人工執行自動的操作。<br>

```go
// 乙獲知甲在B鏈上的地址，實施鏈外約定的任何交易。
```
