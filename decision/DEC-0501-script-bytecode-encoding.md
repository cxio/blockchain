# DEC-0501: Script Bytecode Encoding（脚本字节码编码）

Status: Accepted

## Context（背景）

Conception 明确脚本由 1 字节 opcode、附参和关联数据构成，并定义源码显示约定。附参编码、多附参顺序、关联数据长度、源码字面量歧义和参数消费顺序需要冻结。

## Decision（决策）

建议字节码规则：

- 每条指令以 `opcode byte` 开始。
- 固定宽度附参按 DEC-0001 的大端序编码。
- 变长附参按 DEC-0001 的规范 varint 编码。
- 多个附参按文档声明顺序连续编码，不插入分隔符。
- 关联数据按指令自己的附参长度读取，不额外添加长度前缀。
- 解析器必须拒绝尾随未被任何指令编码消费的残余字节、截断附参和超长关联数据。

建议参数消费顺序：

- 实参区为 FIFO 队列。
- 数据栈弹出为 LIFO。
- 指令需要多个实参时，按文档参数列表从左到右绑定。
- 若从数据栈取多个实参，最后压栈的值绑定到最右侧参数。
- 多返回值按文档返回列表从左到右压入目标区域。

建议源码编译：

- 源码只是显示和编译入口，不参与链上共识。
- 单引号字面量按 Unicode 码点解析；码点 `<=0xff` 时强制编译为 `Byte`，码点 `>0xff` 时编译为 `Rune`。
- 如需将 ASCII 字符以 `Rune` 类型存储，必须使用明确的 Rune 构造或转型机制，不能依赖单引号字面量。
- `BigInt` 的正负数字节编码引用 `DEC-0001` 的 `BigIntBytes = slen || magnitude`。
- 文本别名必须编译为唯一 opcode 和附参，不得保留名称。

`CALL` 公共验证分析：

- `CALL` 不属于正式主网前期禁用指令，但公共验证必须能确定其目标和成本 profile。
- `CALL` 成本估计以目标对象类型和方法名的注册成本 profile 为主。
- `CALL[argc]` 的附参只用于参数读取数量和次级成本计算。
- 公共验证中的 `CALL` 目标对象类型和方法名必须来自字面值或其它静态可追踪的常量路径。
- 公共验证实际执行路径触达 `CALL` 时，目标对象类型、方法名或成本 profile 不能静态解析则验证失败。
- 未执行分支中的 `CALL` 不触发目标解析失败。

## Rationale（理由）

字节码解析必须唯一，否则同一脚本可能拥有多个编码。参数消费顺序会影响几乎所有脚本执行结果，应在实现前冻结。`CALL` 的成本主因来自被调用的目标对象和方法，而不是单纯的参数数量；因此公共验证必须能静态定位调用目标并查到固定成本 profile。

## Consequences（影响）

- 脚本 hash、交易 hash 和脚本成本均基于字节码而非源码。
- 任何禁用指令仍可有 opcode 定义；公共验证实际执行路径触达禁用指令时拒绝执行。
- `CALL` 等半确定性参数数量指令需要额外分析规则，以保证公共验证目标和成本可确定。
- `END` 或公共路径中的 `INPUT` 之后的私有路径仍必须是可解析的字节码，但公共验证不执行、不计成本，也不因其中的禁用指令拒绝。
