# 共识：端点约定

在P2P的网络里，不同于传统「客户端/服务器」的主从逻辑，相互连接的节点是一个平等的关系：一个节点从对方获取服务的同时，也为对方提供服务。节点既是客户端也是服务器，两端依靠预先定义的规则交互和协作。

这是一个由契约维系的去中心化世界，在这里，所有的端点都遵循共同的规则或约定，这些规则和约定，统称为 **端点约定**。

端点约定有两种：

- **协议**：需严格遵守，违背即非法。有最终的合法性验证手段，不遵守会破坏系统，视为错误。
- **共约**：宽松的约束，靠自觉遵守。无需验证也无法验证，如果大家都遵守会很好，少数不遵守也不会致命。

共约没有强制力。但为什么不遵守呢？如果没有不遵守的利益驱动，应该就不是一个问题。

> **注**：在下文中，节点有时也称为端点，不必刻意区分。


## 区块

### 出块时间

铸造者的竞争和优选只与历史交易有关，这与Bitcoin不同。因此出块时间间隔可以设计为一个固定的值。综合权衡，**6分钟** 可能是合适的。

从创始块开始，每一个区块的时间戳都通过高度和时间间隔计算出来（精确值），而不是铸造者节点自己本机的实际时间。


### 区块铸造

当出块时间将要到达时，理论上择优池中的全部候选者都可以出块，但因为区块竞争的条件很明确，通常只需要前端候选者出块就可以了。

节点对最终区块的认可除了验证区块的合法性外，该铸造者还必须是择优池中的成员。


### 铸造冗余

区块的铸造者通常就是择优池中的第一名，但网络世界可能存在任何异常，如果第一名候选者正好离线了，区块铸造需要正常进行。

所以这里的规则是：

*择优池中所有的候选者都可以打包出块，但按顺序执行，间隔 `15秒` 广播*。

当然，如果一名候选者已经收到了排名靠前者打包的区块，且验证通过，那他就不用再发布自己的区块了。

这样，在网络中现实传输的区块并不会太多，因为15秒的广播延迟应该足以阻止后续不必要的发布了。这一延迟考虑了区块广播到全网的时间，以及其它节点对交易差异部分的获取和验证时间。当然，这是一个粗略权衡值。

> **注：**
> 新区块的构建仅依赖前一区块的ID、当前交易集和UTXO/UTCO集指纹。

为了尽可能地收集交易，首个区块将按标准出块时间点（区块时间戳）延后**30**秒发布。即：区块时间戳是标准点时间，在此时间之后30秒才开始发布。


### 区块发布

区块不收录未来的交易（即时间戳*在区块时间戳之后*的交易）。

假设用户都用真实的时间创建并发布交易，在适当的延时后，这些交易应当都已抵达全网，铸造者没有理由等待更久。新区块会在预定的时间发布。

区块的广播和区块内容的验证并不需要同时完成，只要适当的区块证明抵达并验证通过，即可快速转播。

具体策略可能是这样：

1. 如果区块的Coinbase合法，且由局部哈希校验树生成的区块头合法，即先行转播。**注**：铸造者已在择优池中验证过。
2. 同步区块打包的交易ID序列，如果其中有自己缺失的交易，即时请求并验证它们……
3. 最后对完整的交易ID序列计算哈希校验树，构造区块头对比，完成整个区块的合法性验证。

注意：在完整的合法性判断之前，择优池中的候选者*不应当停止*发布区块。


### 区块竞争

合法的区块可能有多个，它们之间的竞争规则如下：

1. 对于同一铸造者创建的区块，收益低的区块胜出。这可以避免铸造者多签的问题。
2. 对于择优池成员冗余发布的区块，有一个交易量约束规则，详见「组队校验」的[交易量约束](附.组队校验.md#交易量约束)部分。

**判断逻辑**：
按上面的顺序，先决定同一铸造者的唯一区块；然后按交易量约束（币权销毁&交易量）对比区块。



## 分叉

通过择优池预选和同步，铸造候选者有了很好的确定性，但这并不能完全保证区块链不出现分叉。如果因为某种原因出现了分叉，比如网络分区或某种攻击，就需要一种规则来决出胜负，找出主链。


### 概念

- **主链**：当前正在运行的，以及未来一直会被认可的、唯一的链。
- **本链**：节点当前正在运行的链，通常就是主链，但也可能是一个分叉。
- **当前区块**：节点当前正在收集交易，待创建的区块。
- **末端区块**：区块链当前末端的既成区块，即当前区块的前一个区块。
- **分叉长度**：节点第一次发现分叉时，分叉末端区块到与本链的分叉点之间的区块数量。
- **临界点**：当状态发生改变时，若网络上的节点很难统一状态，即为临界点。此时很可能出现全网状态割离。


### 分叉竞争

节点间的交互中会包含当前区块的ID，如果同一个高度上有不同的ID，就是出现分叉了。


#### 链段的竞争

不同于Bitcoin的最长链竞争，本设计是固定的出块时间间隔，因此每一条分叉都是等长的。为了体现原始主链的竞争力，这里采用**限定长度链段**的整体评比来实现。

**规则：**

- 分叉之后，链段成长到**31**个区块之前，两条链进行竞争。
- 节点逐块评比两条链段的铸凭哈希，如果某分叉已经胜出过半（16个区块），即切换为本链，提前结束。
- 当节点发现新的分叉时，分叉长度必须小于等于20，否则不予接收。即留出11个区块的时间段用于核实新分叉。

- 如果长度20的分叉刚好出现在其21号区块铸造时间点，则会面临一个临界状况：它是否已经超限？这将由本链当前区块的择优池裁决（详见后）。
- 未被接纳参与评比的分叉会被无视（即便更优），这是一种**既成事实**的社会性共识逻辑。
- 竞争结束后，胜出的分叉正式成为主链并作为节点的本链。

> **注：**
> 区块的铸凭哈希在区块的Coinbase交易里，可由签名验证。
> 铸凭哈希的签名者也是区块头中 CheckRoot 的签名者，二者必须一致。

**31**个区块长度着眼于增加攻击难度：因为参与铸凭哈希计算的信息都是公开的，攻击者可以分析哪些优质铸凭哈希没有参与择优池竞争，由此可以收集并收买、贿赂他们来参与分叉攻击。

但攻击者最多能收买8个区块的铸造者（评参区块为-8号），之后评参区块就已经位于攻击者自己的分叉上了。

设计为16个区块决胜，攻击者仅能收买*50%*的铸造者。虽然这个比例不低，但收买本身也有成本，且仅能用于分叉攻击。如果大额交易的用户等到31个区块之后才认可交易，那么攻击的动机就很低了。

**20**个区块长度主要考虑网络分区的容忍度（2个小时），并为分叉上区块的验证留出足够的时间窗口。


#### 分叉进入裁决

对于分叉长度是20个区块的支链，如果是临近区块铸造时间点才发现，则网络可能割离：部分节点接受而部分不接受。

因此作如下规定：对于长度20的分叉，节点本链当前区块（#21）的择优池成员需进行签名确认：

- 择优池前5名成员自由决定是否接纳该分叉（是或否），签名并广播。
- 节点收集广播的签名消息：按择优池成员顺序，最靠前的成员的裁决有效。
- 如果5名成员都未执行签名确认，则视为默认否决。


#### 平局的可能性及解决

如果不考虑 2^256 规模碰撞的可能性（基本上可以忽略），还有一种情况可能导致两条链段出现平局：*如果同一铸造竞争者同时在两条分叉上参与择优池，且都是实际铸造者。*

此时就可能出现两条竞争链平局的情况。这比 2^256 规模碰撞的可能性大得多，因此需要一个 `tie-breaker` 机制来解决这个问题。

**平局**情况下的分叉解决：

```go
// 哈希算法：
// 选用抗 ASIC 的 `RandomX`，抑制哈希优化动机。
// 参考：github.com/tevador/randomx-go
// 平局的可能性本来就不高，这种简单的抑制应该已经足够。
seed := 分叉点区块ID
input1 := 分叉1首块ID
hash1 := RandomX(seed, input1)
input2 := 分叉2首块ID
hash2 := RandomX(seed, input2)

// 按字典序简单比较即可。
if hash1 < hash2 {
    // 分叉1胜出
} else if hash1 > hash2 {
    // 分叉2胜出
} else {
    // 若相等，取分叉首块ID对比，小者胜出
}
```

如果同高度同铸造者有多个签名的区块，取低收益的区块参与对比。

> **附注：**
> 对于同高度同铸造者的多签区块，低收益区块胜出。
> 这一机制可以抑制铸造者的多签行为，因此实际上分叉竞争平局的情况应该极少。


#### 安全性

分叉评比仅针对末端的31个区块，超出长度的分叉不再理会，这实际上会带来一种确定性安全：

*如果一笔交易已经被31个以上区块确认，那它就基本没什么可担忧的了*。

> **注：**
> 在这一机制下，合理的分叉是允许的，但它们属于自由的第三方支链，与本链无关。


### 交易回收

分叉竞争结束后，失败分叉上的交易应当合并到主链上来。

如果由网络分区导致分叉，其不同的交易应该会很多，但如果是攻击导致的分叉，不同的交易应该会很少。但无论如何，失败分叉上的交易都应该被回收进入主链，以免除用户的担忧。

作为一个共约，新币通常在拥有*31*个区块确认之后才会被使用，所以无效的新币输入源应该不会太多，排除即可。分叉会导致过早使用新币的交易无效，这是用户应当警惕的风险。

> **设计：**
> 新币31确认后才使用是一个共约而非协议，这是一种友好性权衡。



## 其它

### 关于零确认

交易在网络上的传输没有先后顺序，原则上，交易一旦广播出去便无法撤回。

如果一位用户发送了一笔失误的交易（比如接收者错误），应当允许他立即重新发送一笔正确的交易，但这就出现了双花。通常，铸造者倾向于收录交易费更高的交易，所以如果用户要纠正失误，应当在区块打包前发送且提供更高一些的交易费。

但双花也可能源于攻击。如果接收者认可零确认，在收到前一笔交易后发货或让顾客离开，而最终进入区块的却是另一笔交易，这就导致收款者的损失了。

从系统上设计零确认的安全机制会比较复杂，这里只作简化处理：*如果一个节点发现双花交易，应在当前App上发送警告提醒用户*。

因此对于大笔金额或十分重要的交易，用户应当等待足够的区块确认。


### 最低交易费

交易费是矿工的重要收入来源，提供服务收取费用是一种正常的逻辑。

区块收录的交易数量是有限的，交易的入块竞争可能带来交易费的竞争。虽然如此，一个恰当的最低交易费约束是有益的，它可以避免为铸造资格而“刷交易”的成本不至于太低。

> **警告：**
> 节点不应当阻止低交易费交易的转播，因为铸造者可能已经收录了它。
> 这种阻止会导致部分节点无法验证该区块是否合法。

这是一个**共约**，是否收录低于最低交易费的交易，决定权在铸造者本身。


### 错时延迟

当一笔交易传播至节点时，交易的时间戳早于当前节点的实际时间太多，就称这是一笔**错时交易**。

区块可以收录错时的交易（它们属于过去），但这些错时交易可能成为铸造者贪婪的诱因，影响他们按时出块，所以这里设计了一个约定：

*当出块时间到达后，时间戳在出块时间之前的错时交易会被暂停转播，直到该区块被创建、广播并确定下来*。

这就是「错时延迟」共约。对于区块铸造者来说，时间到了，就没有可打包的新交易传输过来了，所以没必要再等待，按时出块即可。


### 交易过期

未被收录的交易超过一定时间后会被作废，这个期限设计为24小时（`240个` 区块）。

过期是按交易的时间戳判断的，它的意义在于：

1. 缩减未确认交易的规模。
2. 提升时间因子的价值，为某些应用提供一种*确定性*。

这是一个硬性协议，公共验证节点需要严格遵守。

> **注记：**
> 人们不应当期待一笔24小时都未确认的交易依然有效，长时等待其实体验负面。



## 设计参数

- 区块的出块时间间隔为 `6分钟`，每一个区块的时间戳都由创始块的时间戳计算而来。
- 择优池中的候选者按顺序发布区块，时间间隔为 `15秒`，首个区块按标准时间点延后 `30秒` 出块。
- 分叉必须成长到 `31个` 区块（约3小时）后评比才结束。这也是新币花费的安全确认数。
- 最低交易费由前期 `6000个` 区块的平均交易费计算而来，取其 `1/4` 作为最低费用。这是一个**共约**。
- 超出 `240个` 区块都未收录的交易视为过期失效。这是一个*协议*，不可违背。即铸造者不能收录这样的交易。
